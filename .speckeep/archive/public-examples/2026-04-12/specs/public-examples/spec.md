# Публичные production-ready примеры в README

## Scope Snapshot

- In scope: добавить в `README.md` 1–2 коротких end-to-end примера “production-ready” (Pipeline + store + embedder + кеш/ретраи) с рекомендуемыми таймаутами и корректным использованием `context.Context`.
- Out of scope: изменение публичного API библиотеки, добавление новых провайдеров/хранилищ, редизайн структуры `examples/` и CI.

## Цель

Новые пользователи draftRAG должны видеть в `README.md` компактные, копипастабельные примеры, которые демонстрируют “правильную” сборку пайплайна для production: осмысленные таймауты, retry/circuit breaker, кэширование эмбеддингов и подключение реального store (pgvector или Qdrant).

## Основной сценарий

1. Пользователь открывает `README.md` и ищет пример “как делать правильно в production”.
2. Пользователь выбирает пример под свой store (pgvector или Qdrant) и копирует код.
3. Пользователь подставляет переменные окружения/DSN и запускает пример.
4. При временных сбоях внешних API (embedder/LLM) пример показывает корректную стратегию retry/CB и предсказуемые таймауты через контекст.

## Scope

- Добавить в `README.md` новый подраздел с 1–2 end-to-end примерами, ориентированными на production.
- Примеры ДОЛЖНЫ использовать существующие публичные типы/фабрики из `pkg/draftrag` и демонстрировать:
  - `context.WithTimeout` для ключевых операций;
  - `NewCachedEmbedder` (L1/L2 кэш при необходимости) поверх базового embedder’а;
  - `NewRetryEmbedder` и/или `NewRetryLLMProvider` с безопасными defaults или явно заданными опциями;
  - создание `Pipeline` с реальным `VectorStore` (pgvector через `NewPGVectorStore*` или Qdrant через `NewQdrantStore`).
- Примеры должны оставаться небольшими: без “фреймворка приложения”, без лишних абстракций, с минимальными зависимостями.

## Контекст

- Проект — Go-библиотека; минимальная версия Go: 1.23.
- По конституции все операции используют `context.Context` и должны демонстрировать контекстную безопасность.
- `README.md` уже содержит “быстрый старт” и примеры отдельных возможностей; цель этой фичи — добавить именно production-ориентированные end-to-end примеры (таймауты/ретраи/кеш).
- В репозитории уже есть каталог `examples/` (pgvector/qdrant); README-примеры можно согласовать с ними, чтобы не расходиться по API.

## Требования

- RQ-001 `README.md` ДОЛЖЕН содержать 1–2 новых end-to-end примера “production-ready” с явным wiring: `VectorStore` + `Embedder` + `LLMProvider` + `Pipeline`.
- RQ-002 Каждый новый пример ДОЛЖЕН демонстрировать рекомендуемое использование `context.Context`: как минимум `context.WithTimeout` для индексации и для запроса/ответа.
- RQ-003 Каждый новый пример ДОЛЖЕН включать resilience-слой: retry/circuit breaker для embedder и/или LLM (через публичные обёртки `NewRetryEmbedder` / `NewRetryLLMProvider`).
- RQ-004 Каждый новый пример ДОЛЖЕН включать кэширование эмбеддингов через публичный `NewCachedEmbedder` (in-memory LRU), а при наличии места — кратко показать, как подключить Redis L2 (без привязки к конкретной библиотеке Redis).
- RQ-005 Примеры НЕ ДОЛЖНЫ требовать изменения публичного API draftRAG; используются только существующие экспортируемые идентификаторы.

## Вне scope

- Добавление/изменение публичных API, новых опций, новых пакетов или breaking changes.
- Переписывание существующих примеров в `examples/` или перенос их структуры.
- Добавление “полноценного” production-референс приложения (HTTP-сервер, конфиг-менеджмент, контейнеризация).

## Критерии приемки

### AC-001 README содержит production-ready pgvector пример

- Почему это важно: pgvector — типичный production-выбор; пример снижает время до первого корректного внедрения.
- **Given** пользователь читает `README.md`
- **When** пользователь открывает новый подраздел “Production-ready” и выбирает pgvector пример
- **Then** он видит короткий end-to-end код, который включает: `NewPGVectorStore*`, `NewCachedEmbedder`, `NewRetryEmbedder`/`NewRetryLLMProvider`, `NewPipeline*`, и демонстрацию таймаутов через `context.WithTimeout`
- Evidence: в `README.md` присутствует отдельный код-блок pgvector, и он использует только экспортируемые символы `pkg/draftrag`

### AC-002 README содержит production-ready Qdrant пример (или альтернативный второй пример)

- Почему это важно: пользователи выбирают разные сторы; второй пример показывает переносимость паттерна.
- **Given** пользователь читает `README.md`
- **When** пользователь выбирает второй production-ready пример
- **Then** он видит end-to-end wiring для Qdrant (через `NewQdrantStore` и/или helper’ы коллекций) с теми же практиками: таймауты, retry/CB, кэш эмбеддингов
- Evidence: в `README.md` есть второй код-блок, который компонуется вокруг публичного API и не вводит новые зависимости проекта

### AC-003 Примеры задают “рекомендуемые” таймауты и контекст без двусмысленности

- Почему это важно: “production-ready” без таймаутов и корректного контекста вводит в заблуждение.
- **Given** пользователь копирует пример
- **When** он читает секцию про контекст/таймауты в примере
- **Then** таймауты заданы явно (с разумными значениями по умолчанию) и ясно привязаны к операциям (индексация vs поиск/ответ), без `context.Background()` как единственного контекста для всех вызовов
- Evidence: в коде примера есть `context.WithTimeout(...)` и defer cancel, а значения таймаутов указаны рядом и документированы одной строкой

## Допущения

- Текущий публичный API `pkg/draftrag` уже содержит: фабрики `NewCachedEmbedder`, `NewRetryEmbedder`, `NewRetryLLMProvider`, `NewPGVectorStore*`, `NewQdrantStore`, и конструкторы `Pipeline`.
- README допускает добавление новых подразделов и код-блоков на русском языке (текущий язык README — русский).
- “Рекомендуемые таймауты” в рамках README являются ориентиром и не претендуют на универсальность для всех окружений.

## Краевые случаи

- Ошибки контекста (`context.DeadlineExceeded`, `context.Canceled`) не должны быть представлены как retryable: пример должен показывать корректный порядок `defer cancel()` и обработку ошибок.
- Внешние API могут быть временно недоступны: пример должен явно показывать включение retry/CB (даже если опции оставлены дефолтными).
- Кэш эмбеддингов должен корректно работать при высоком повторе одинаковых запросов: пример должен показывать, что кэш — слой поверх базового embedder’а.
