# OpenTelemetry observability (Hooks)

## Scope Snapshot

- In scope: добавить опциональную OpenTelemetry-интеграцию для наблюдаемости стадий pipeline draftRAG через существующий интерфейс `draftrag.Hooks`, плюс короткий пример в README.
- Out of scope: автоконфигурация OTel SDK/экспортеров, готовые дашборды, трассировка сетевых клиентов/БД и “магическая” интеграция без явного подключения hooks пользователем.

## Цель

Команды стартапов и энтерпрайза должны получить стандартный, легко подключаемый способ собирать трассировки и метрики по стадиям RAG-пайплайна (chunking/embed/search/generate) через OpenTelemetry, чтобы быстрее диагностировать latency/ошибки и вводить SLO без форка библиотеки.

## Основной сценарий

1. Пользователь настраивает OpenTelemetry SDK у себя в приложении (tracer/meter/exporter).
2. Пользователь подключает OTel hooks в `draftrag.PipelineOptions{Hooks: ...}`.
3. При индексации и ответах pipeline генерирует spans и метрики по стадиям.
4. При ошибках стадии отражаются как error-status в span и увеличивают счётчики ошибок.

## Scope

- Добавить новый публичный (опциональный) пакет/поверхность для OTel hooks, который **реализует `draftrag.Hooks`** и использует `context.Context` для propagation.
- Интеграция ДОЛЖНА покрывать как минимум стадии, уже отражённые в hooks: `chunking`, `embed`, `search`, `generate`, и операции, которые уже передаются в `StageStartEvent.Operation`.
- Добавить короткий пример в `README.md`: как подключить hooks в `PipelineOptions` и какие атрибуты/метрики ожидать.

## Контекст

- В `internal/domain/hooks.go` определён минимальный синхронный интерфейс `Hooks` (StageStart/StageEnd) и фиксированный набор стадий.
- Hooks вызываются синхронно; обработчик должен быть лёгким (иначе пользователь сам создаст latency).
- В `pkg/draftrag` `Hooks` переэкспортируется, поэтому интеграция должна жить на публичной поверхности и быть совместимой с “чистым” использованием библиотеки.

## Требования

- RQ-001 Библиотека ДОЛЖНА предоставить OTel-реализацию `draftrag.Hooks`, пригодную для подключения в `PipelineOptions.Hooks`.
- RQ-002 OTel hooks ДОЛЖНЫ создавать span на каждую стадию (chunking/embed/search/generate) с атрибутами `operation` и `stage`, и помечать ошибки (если `StageEndEvent.Err != nil`).
- RQ-003 OTel hooks ДОЛЖНЫ экспортировать как минимум одну метрику длительности стадий и одну метрику ошибок стадий (labels: `operation`, `stage`).
- RQ-004 Интеграция ДОЛЖНА быть опциональной для пользователя: без подключения hooks поведение pipeline не меняется.
- RQ-005 Документация ДОЛЖНА описывать ограничения синхронных hooks и рекомендовать неблокирующие exporters/SDK настройки (на уровне README).

## Вне scope

- Автоматическое поднятие OTel SDK (инициализация exporter, resource, sampler, collector endpoint).
- Полная observability для всех внутренних операций (HTTP клиенты провайдеров, SQL запросы, retries/circuit breaker события).
- Готовые дашборды (Grafana), алертинг и конкретные SLO значения.

## Критерии приемки

### AC-001 Публичный OTel hooks доступен и подключаем в PipelineOptions

- Почему это важно: пользователи должны подключать наблюдаемость без форка и без трогания internal.
- **Given** пользователь использует `github.com/bzdvdn/draftrag/pkg/draftrag`
- **When** он ищет способ подключить OTel наблюдаемость по стадиям
- **Then** он находит публичный конструктор/тип OTel hooks, который удовлетворяет `draftrag.Hooks` и передаётся в `draftrag.PipelineOptions{Hooks: ...}`
- Evidence: новый публичный пакет/тип/конструктор виден в godoc и может быть использован как `Hooks`

### AC-002 Трейсинг стадий работает и отражает ошибки

- Почему это важно: без spans невозможно быстро локализовать bottleneck по стадиям.
- **Given** пользователь подключил OTel SDK и передал OTel hooks в pipeline
- **When** он выполняет `Index` или `Answer` и одна из стадий завершается ошибкой
- **Then** в трассировке появляется span для стадии с атрибутами `operation` и `stage`, а ошибка отражена как error-status/recorded error
- Evidence: при запуске теста/примерного приложения в exporter видны spans по стадиям и помеченные ошибки

### AC-003 Метрики по стадиям доступны для алертинга/дашбордов

- Почему это важно: метрики нужны для SLO, алертов и долгосрочных трендов.
- **Given** пользователь подключил OTel Meter и передал OTel hooks в pipeline
- **When** он выполняет операции pipeline
- **Then** он получает метрики длительности стадий и счётчик ошибок стадий с лейблами `operation` и `stage`
- Evidence: в метрик-экспортере/скрейпе присутствуют указанные метрики и корректные labels

### AC-004 README содержит короткий пример подключения OTel hooks

- Почему это важно: без примера пользователи будут делать “свой” формат и терять совместимость.
- **Given** пользователь читает `README.md`
- **When** он хочет быстро включить OTel наблюдаемость
- **Then** он видит компактный пример подключения hooks в `PipelineOptions` и краткое описание ожидаемых span/metric атрибутов и ограничения синхронных hooks
- Evidence: в README есть новая секция/подсекция про OTel hooks

## Допущения

- Проект может принять зависимость `go.opentelemetry.io/otel/*` как часть модуля, при этом интеграция остаётся опциональной по использованию.
- Текущего payload hooks (`operation`, `stage`, `duration`, `err`) достаточно для v1 observability; расширение payload (добавление новых полей) — отдельная фича.

## Критерии успеха

- SC-001 Пользователь может увидеть p95 latency по стадиям (embed/search/generate) в метриках без модификации кода библиотеки.

## Краевые случаи

- Высокая нагрузка: hooks не должны существенно увеличивать latency; в docs указать, что exporters должны быть неблокирующими/батчевыми.
- `context.Canceled`/`context.DeadlineExceeded`: ошибки должны отражаться как ошибки стадии, но hooks не должны пытаться “лечить” контекст.
- Наличие нескольких pipeline в одном процессе: labels/атрибуты должны позволять агрегировать по `operation` и `stage` без глобальных синглтонов.
