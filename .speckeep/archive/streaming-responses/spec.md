# Streaming ответов

## Scope Snapshot

- In scope: Добавление streaming-генерации ответов токен за токеном через канал (<-chan string) для улучшения UX в web/CLI приложениях.
- Out of scope: WebSocket сервер, Server-Sent Events endpoint, интеграция с конкретными фреймворками (React, Vue и т.д.).

## Цель

Разработчики, использующие draftRAG, получат возможность стримить ответы от LLM в реальном времени. Это критично для UX — пользователь видит текст по мере генерации вместо ожидания полного ответа. Успех измеряется возможностью получать токены через канал без потери существующего функционала.

## Основной сценарий

1. Разработчик инициализирует Pipeline с LLMProvider, поддерживающим streaming.
2. Разработчик вызывает `AnswerStream(ctx, question, topK)`, получая канал `<-chan string`.
3. Приложение читает из канала и отображает каждый токен/чанк по мере получения.
4. Канал закрывается при завершении генерации или ошибке (ошибка передаётся через отдельный возвращаемый error).
5. Для inline-цитирования доступен `AnswerStreamWithInlineCitations` с аналогичным поведением.

## Scope

- Добавление `GenerateStream()` в интерфейс `LLMProvider` (или отдельный capability-интерфейс `StreamingLLMProvider`)
- Реализация SSE/chunked parsing для OpenAI-compatible LLM
- Добавление `AnswerStream`, `AnswerStreamWithInlineCitations` в публичный API (`pkg/draftrag`)
- Корректная обработка отмены через `context.Context`
- Мок-реализация для тестирования

## Контекст

- Существующий `LLMProvider` имеет только синхронный метод `Generate`
- OpenAI Responses API поддерживает streaming через `stream: true` параметр
- Канал Go (`<-chan string`) — идиоматичный способ стриминга данных
- Необходимо сохранить backward compatibility — существующие реализации `LLMProvider` должны продолжать работать
- Чистая архитектура требует: domain-интерфейс → application-use-case → infrastructure-реализация → public API

## Требования

- RQ-001 Интерфейс `StreamingLLMProvider` ДОЛЖЕН быть добавлен как capability-расширение `LLMProvider` с методом `GenerateStream(ctx context.Context, systemPrompt, userMessage string) (<-chan string, error)`
- RQ-002 Реализация OpenAI-compatible LLM ДОЛЖНА поддерживать streaming через SSE (Server-Sent Events) с корректным парсингом `data:` линий
- RQ-003 Публичный Pipeline ДОЛЖЕН предоставлять метод `AnswerStream(ctx context.Context, question string, topK int) (<-chan string, error)`
- RQ-004 Публичный Pipeline ДОЛЖЕН предоставлять метод `AnswerStreamWithInlineCitations(ctx context.Context, question string, topK int) (<-chan string, []InlineCitation, error)` — канал для текста + финальный маппинг цитат
- RQ-005 При отмене контекста канал ДОЛЖЕН закрываться и горутина-продюсер ДОЛЖНА завершаться без утечек
- RQ-006 При ошибке streaming'а канал ДОЛЖЕН закрываться и ошибка ДОЛЖНА быть возвращена через error return value
- RQ-007 Мок-реализация `StreamingLLMProvider` ДОЛЖНА быть доступна для тестирования

## Вне scope

- Реализация streaming для других LLM-провайдеров (Anthropic, Ollama) — только OpenAI-compatible
- WebSocket или HTTP сервер для streaming — draftRAG остаётся библиотекой без встроенного сервера
- Интеграция с конкретными frontend-фреймворками
- Streaming для методов без retrieval (чистая генерация без контекста)
- Дедупликация или MMR для streaming-ответов (используется существующая логика retrieval)

## Критерии приемки

### AC-001 Streaming генерация через канал

- Почему это важно: Пользователь видит ответ в реальном времени без ожидания полной генерации
- **Given** Pipeline инициализирован с `OpenAICompatibleLLM`, который поддерживает streaming
- **When** вызывается `AnswerStream(ctx, "вопрос", 5)`
- **Then** возвращается `<-chan string`, из которого можно читать токены/чанки до закрытия канала
- Evidence: Можно написать тест, который собирает все значения из канала и проверяет, что полученный текст совпадает с ожидаемым

### AC-002 Streaming с inline-цитатами

- Почему это важно: Пользователь получает обогащённый ответ с цитатами в реальном времени
- **Given** Pipeline инициализирован с `OpenAICompatibleLLM`
- **When** вызывается `AnswerStreamWithInlineCitations(ctx, "вопрос", 5)`
- **Then** возвращается канал для текста + слайс `[]InlineCitation`, который заполняется при завершении
- Evidence: Тест проверяет, что канал возвращает текст, а финальный `[]InlineCitation` содержит валидные цитаты

### AC-003 Обработка отмены контекста

- Почему это важно: При отмене запроса (например, пользователь закрыл окно) не должно быть утечек
- **Given** `AnswerStream` вызван и streaming активен
- **When** `ctx.Cancel()` вызывается
- **Then** канал закрывается в разумное время (< 100мс), горутина завершается, нет паник
- Evidence: Тест с `context.WithCancel()` + `go leak detector` или простая проверка на deadlock

### AC-004 Backward compatibility

- Почему это важно: Существующий код должен продолжать работать без изменений
- **Given** Существующая реализация `LLMProvider` без `GenerateStream`
- **When** Pipeline инициализируется с этой реализацией
- **Then** Методы `Answer*` продолжают работать; `AnswerStream*` возвращают ошибку "streaming not supported"
- Evidence: Существующие тесты проходят без изменений; новый тест проверяет graceful degradation

### AC-005 OpenAI-compatible streaming парсинг

- Почему это важно: Корректная работа с реальным OpenAI API (или совместимыми)
- **Given** LLM настроен на реальный или мок OpenAI endpoint с `stream: true`
- **When** `GenerateStream` вызывается
- **Then** Канал получает валидные текстовые чанки из SSE `data:` линий, игнорируя `: ping` и пустые линии
- Evidence: Интеграционный тест с мок-сервером, проверяющий парсинг SSE формата

## Допущения

- OpenAI Responses API или совместимый API поддерживает `stream: true` параметр и SSE формат
- Streaming возвращает текстовые дельты, которые можно конкатенировать для получения полного ответа
- Задержка между токенами не критична для первой версии (не требуется агрессивная буферизация)
- Пользователь приложения отвечает за преобразование канала Go в web-формат (SSE, WebSocket и т.д.)
- Go 1.21+ поддерживает все необходимые примитивы для конкурентного программирования

## Критерии успеха

- SC-001 Streaming ответ для типичного вопроса (100 токенов) начинается за < 500мс от отправки запроса
- SC-002 Меморик-профиль: при активном streaming не более одной копии ответа в памяти (используется канал, не буферизация всего ответа)

## Краевые случаи

- Пустой ответ от LLM: канал закрывается без отправленных значений, error = nil
- LLM возвращает ошибку в середине streaming'а: канал закрывается, error содержит описание ошибки
- Очень быстрый ответ (1-2 токена): канал получает 1-2 значения и закрывается
- Нестабильное соединение: зависит от HTTPClient timeout и context deadline
- Вложенные цитаты в streaming с inline citations: цитаты собираются по мере генерации и возвращаются в финальном слайсе
