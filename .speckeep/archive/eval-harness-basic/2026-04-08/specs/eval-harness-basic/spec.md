# Eval harness: базовая оценка качества RAG (v1)

## Scope Snapshot

- In scope: добавить библиотечный eval harness для оффлайн-оценки retrieval/answer качества на датасете (без сетевых зависимостей в тестах).
- Out of scope: CLI, сложные метрики faithfulness с LLM-оценкой, автоматическая генерация датасетов.

## Контекст

- Сейчас изменения в retrieval (например: дедупликация, ранжирование) и в prompt могут заметно менять качество ответа, но это тяжело поймать автоматически.
- Нужен простой, детерминированный (в тестах) способ прогнать набор кейсов и получить базовые retrieval-метрики, чтобы:
  - сравнивать варианты реализации,
  - предотвращать регрессии в CI,
  - иметь “числа” для обсуждения качества.

## Цель

Сделать качество измеримым и предотвратить регрессии: при изменениях retrieval/prompt можно прогнать набор кейсов и сравнить метрики.

## Основной сценарий

1. Пользователь задаёт датасет кейсов: question, ожидаемые источники (или doc ids), optional expected answer.
2. Harness прогоняет pipeline (с fake или реальными провайдерами в проде).
3. Возвращает метрики (например hit@k, mrr) и подробный отчёт по кейсам.

## Требования

- RQ-001 ДОЛЖНЫ быть определены модели данных для eval-case и eval-result (в pkg или internal, но доступно для пользователя).
- RQ-002 Harness ДОЛЖЕН поддерживать минимум retrieval-метрики:
  - hit@k по `ParentID` (или ID чанка),
  - MRR (mean reciprocal rank).
- RQ-003 Harness ДОЛЖЕН быть детерминированным при одинаковых входах (для fake providers).
- RQ-004 Тесты ДОЛЖНЫ проходить без внешней сети.
- RQ-005 Добавление harness НЕ ДОЛЖНО ломать публичный API.

## Критерии приемки

### AC-001 Метрики считаются корректно на синтетическом датасете

- **Given** небольшой датасет с известными “правильными” источниками
- **When** прогоняется harness
- **Then** hit@k и MRR равны ожидаемым значениям

### AC-002 Аддитивность

- **Given** текущая библиотека
- **When** добавляется eval harness
- **Then** `go test ./...` проходит и существующие импорты не ломаются
