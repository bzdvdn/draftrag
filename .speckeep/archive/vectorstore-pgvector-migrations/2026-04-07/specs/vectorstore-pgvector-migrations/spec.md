# VectorStore pgvector: миграции и стабильная схема (v1)

## Scope Snapshot

- In scope: зафиксировать схему pgvector-хранилища в виде миграций (extension, таблицы, индексы), чтобы проект можно было разворачивать и обновлять предсказуемо.
- Out of scope: автоматическое выполнение миграций библиотекой “из коробки”, multi-DB поддержка, сложные миграционные сценарии (sharding, online reindex), инструменты администрирования.

## Цель

Сделать pgvector-бэкенд пригодным для продакшена: схема и индексы должны быть описаны миграциями и воспроизводимо применяться в любой среде.

## Основной сценарий

1. Пользователь подключает PostgreSQL + pgvector.
2. Применяет миграции (внешним инструментом или своим кодом) перед запуском приложения.
3. Использует `pgvector` реализацию `VectorStore` без ручного SQL “в README”.

## Контекст

- В проекте уже есть реализация `VectorStore` на PostgreSQL+pgvector и она должна иметь воспроизводимую схему БД.
- Сейчас схема (таблицы/индексы/extension) может создаваться “вручную” и это усложняет развёртывание, CI и обновления.
- Миграции нужны как единый источник истины для схемы (включая выбранные индексы под similarity search и поддерживаемые фильтры).

## Требования

- RQ-001 ДОЛЖЕН быть набор SQL-миграций для:
  - включения `vector` extension (где это допустимо),
  - создания основных таблиц/индексов для хранения чанков,
  - создания необходимых индексов для similarity search.
- RQ-002 Миграции ДОЛЖНЫ быть детерминированными и идемпотентными там, где это возможно (например `CREATE EXTENSION IF NOT EXISTS`).
- RQ-003 В проекте ДОЛЖНА быть задокументирована минимальная последовательность применения миграций для pgvector.
- RQ-004 Индекс(ы) ДОЛЖНЫ поддерживать выбранную метрику (cosine/ip/l2) и иметь настройки, подходящие для v1 (без тюнинга под каждую нагрузку).
- RQ-005 Тесты ДОЛЖНЫ проходить без внешней сети.

## Критерии приемки

### AC-001 Миграции присутствуют и однозначно применимы

- **Given** пользователь открывает репозиторий
- **When** он ищет миграции для pgvector
- **Then** он находит набор миграций и порядок их применения (без догадок)

### AC-002 Схема покрывает основные операции VectorStore

- **Given** `VectorStore.Upsert/Delete/Search`
- **When** пользователь применил миграции
- **Then** схема содержит необходимые таблицы/индексы для корректной работы этих методов

### AC-003 Backward compatibility

- **Given** существующие публичные API пакеты
- **When** запускаются тесты проекта
- **Then** `go test ./...` проходит (миграции добавлены аддитивно)
