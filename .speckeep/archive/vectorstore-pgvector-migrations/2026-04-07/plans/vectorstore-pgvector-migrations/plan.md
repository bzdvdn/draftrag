# VectorStore pgvector: миграции и стабильная схема (v1) — План

## Phase Contract

Inputs: `.draftspec/specs/vectorstore-pgvector-migrations/spec.md`, `.draftspec/specs/vectorstore-pgvector-migrations/inspect.md`, `.draftspec/constitution.md`.
Outputs: `.draftspec/plans/vectorstore-pgvector-migrations/plan.md`, `.draftspec/plans/vectorstore-pgvector-migrations/data-model.md`.
Stop if: нельзя описать схему в виде миграций без конфликта с текущим API/реализацией pgvector store.

## Цель

Зафиксировать pgvector-схему как воспроизводимый набор миграций (extension + таблицы + индексы) и задокументировать порядок их применения, чтобы pgvector store можно было разворачивать и обновлять предсказуемо.

## Scope

- В scope:
  - SQL-миграции для базовой схемы хранения чанков и индексов, необходимых для `Upsert/Delete/Search` и фильтров.
  - Документация: “какие файлы применить и в каком порядке”.
- Вне scope:
  - Автоматический запуск миграций библиотекой “сам по себе” (без явного вызова пользователем/инструментом).
  - Online reindex / CONCURRENTLY, сложные сценарии миграций и операционные тулзы.

## Implementation Surfaces

- `pkg/draftrag/pgvector.go`, `pkg/draftrag/pgvector_migrate.go`:
  - зафиксировать роль существующих `MigratePGVector` и `SetupPGVector` относительно SQL-миграций (не ломая backward compatibility).
- Новая поверхность: `pkg/draftrag/migrations/pgvector/`:
  - добавить версионированные `.sql` файлы миграций (таблица, индексы, расширение).
- Документация (новая поверхность):
  - файл с краткой инструкцией применения миграций: `pkg/draftrag/pgvector_migrations.md`.
- Тесты:
  - лёгкие unit-тесты (например `pkg/draftrag/pgvector_migrations_assets_test.go`), подтверждающие наличие migration assets и их “однозначность” (список файлов/версий), без подключения к БД.

## Влияние на архитектуру

- Domain не меняется.
- Публичный API не ломается: добавления аддитивны (migration assets + документация), существующие `SetupPGVector/MigratePGVector` остаются.
- Миграции становятся источником истины для схемы; runtime DDL (если остаётся) должен быть согласован с миграциями.

## Acceptance Approach

- AC-001 -> миграции лежат в очевидном месте и есть краткая инструкция применения (docs).
- AC-002 -> миграции создают таблицу и индексы, достаточные для `Upsert/Delete/Search` и фильтрации (по текущим столбцам и индексам).
- AC-003 -> `go test ./...` проходит; изменения только аддитивные.

## Данные и контракты

- Persisted state: таблица чанков + индексы + (опционально) миграционная таблица версий.
- Конфигурационные параметры (например `EmbeddingDimension`, выбор индекса) должны быть отражены в миграциях/документации так, чтобы не требовалось “угадывать” (DEC-002).

## Стратегия реализации

- DEC-001 “SQL-миграции — основной артефакт схемы”
  Why: они воспроизводимы и удобны для деплоя (deploy job / init container).
  Tradeoff: требуется синхронизировать runtime DDL (если есть) с файлами миграций.
  Affects: `pkg/draftrag/migrations/pgvector/*.sql`, документация, (опционально) `MigratePGVector`.
  Validation: наличие миграций + документированный порядок применения (AC-001).

- DEC-002 “Миграции исполняемы без параметров, а вариативность — через документированный playbook”
  Why: чистые SQL-миграции не умеют безопасно параметризоваться под разные `lists/IndexMethod` без выбора инструмента.
  Tradeoff: часть сценариев (смена метода индекса/тюнинг) оформляется как manual playbook, не как автоматическая миграция v1.
  Affects: docs + список поддерживаемых дефолтов (например дефолтный embedding-индекс).
  Validation: миграции применимы “как есть” (AC-001) и покрывают базовый сценарий (AC-002).

## Incremental Delivery

### MVP (Первая ценность)

- Добавить директорию с SQL-миграциями для базовой схемы.
- Добавить короткую документацию “как применить миграции”.
- Добавить минимальный тест на наличие migration assets.

Критерий готовности MVP: AC-001 и AC-003.

### Итеративное расширение

- Расширить миграции под дополнительные поля/индексы (metadata/timestamps), если это часть текущей схемы.
- Добавить playbook для смены embedding-индекса (drop+create) и для метрик (cosine/ip/l2), если поддерживается.

Критерий готовности: AC-002.

## Порядок реализации

1. Определить “источник истины” текущей схемы (какие столбцы/индексы должны быть в v1).
2. Зафиксировать это в SQL-миграциях (версионно).
3. Добавить документацию применения.
4. Добавить тесты-гарды на наличие/версии миграций.
5. Прогнать `go test ./...`.

## Риски

- Риск 1: рассинхронизация между SQL-миграциями и `MigratePGVector`.
  Mitigation: в tasks добавить проверку согласованности (например список версий/DDL snippets) и держать один “канонический” набор изменений.
- Риск 2: разные окружения и права на `CREATE EXTENSION`.
  Mitigation: в docs описать режимы (с правами/без прав), оставить extension как отдельный шаг/миграцию.

## Rollout и compatibility

- Rollout: миграции применяются отдельным шагом деплоя; библиотека не запускает их автоматически.
- Compatibility: только аддитивные изменения; существующие тесты и API должны остаться рабочими.

## Проверка

- `go test ./...` (AC-003).
- Review evidence:
  - миграционные файлы присутствуют и имеют понятные имена/порядок (AC-001),
  - в миграциях есть DDL для таблиц и индексов под `Upsert/Delete/Search` (AC-002).

## Соответствие конституции

- Нет конфликтов: изменения не вводят внешних зависимостей, не ломают API, остаются тестируемыми и контекстно-безопасными.
