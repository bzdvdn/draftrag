# VectorStore pgvector: миграции и стабильная схема (v1) — Модель данных

## Scope

- Связанные `AC-*`: `AC-001`, `AC-002`
- Связанные `DEC-*`: `DEC-001`, `DEC-002`
- Data model описывает persisted state для pgvector store (таблица и индексы). Domain-модели не меняются.

## Сущности

### DM-001 PGVectorChunksTable (таблица чанков)

- Назначение: хранить чанки документов с embedding-векторами для similarity search и фильтрации.
- Источник истины: SQL-миграции `pkg/draftrag/migrations/pgvector/*.sql` (v1).
- Инварианты:
  - `id` уникален и является primary key.
  - `parent_id` обязателен и используется для фильтров и группировки источников.
  - `content` не пустой (на уровне приложения/валидации), хранится как `TEXT`.
  - `position` обязателен и позволяет упорядочивать чанки внутри документа.
  - `embedding` имеет фиксированную размерность (`VECTOR(N)`), согласованную с Embedder.
- Связанные `AC-*`: `AC-002`
- Связанные `DEC-*`: `DEC-001`, `DEC-002`
- Поля (минимум v1):
  - `id` — `TEXT`, required, primary key
  - `parent_id` — `TEXT`, required
  - `content` — `TEXT`, required
  - `position` — `INTEGER`, required
  - `embedding` — `VECTOR(N)`, required
  - (опционально, если уже часть схемы v1): `metadata jsonb`, `created_at`, `updated_at`
- Жизненный цикл:
  - создаётся/обновляется: `Upsert`
  - удаляется: `Delete`
  - читается: `Search`/`SearchWithFilter`
- Замечания по консистентности:
  - недопустимо хранить embedding неправильной размерности (должно быть предотвращено конфигом/валидацией).
  - индексы должны соответствовать реальным запросам (особенно фильтры по `parent_id`).

### DM-002 PGVectorIndexes (индексы для производительности)

- Назначение: обеспечить быстрый `Search` по embedding и быстрые фильтры по `parent_id`.
- Источник истины: SQL-миграции.
- Инварианты:
  - существует embedding-индекс (по выбранному дефолтному методу v1) для `embedding`.
  - существует индекс по `parent_id` (и при необходимости `parent_id, position`) для фильтрации/порядка.
- Связанные `AC-*`: `AC-002`
- Связанные `DEC-*`: `DEC-001`, `DEC-002`
- Поля:
  - `embedding_idx` — `USING ivfflat|hnsw` с выбранной опс-метрикой (v1 дефолт фиксируется в docs)
  - `parent_id_idx` — btree `(parent_id)`
  - `parent_pos_idx` — btree `(parent_id, position)` (если используется приложением)

### DM-003 PGVectorSchemaVersion (учёт версий схемы)

- Назначение: фиксировать применённые версии миграций (для безопасных обновлений).
- Источник истины: SQL-миграции (если таблица используется) и документация.
- Инварианты:
  - version монотонно растёт (новые миграции — новые версии).
- Связанные `AC-*`: `AC-001`
- Связанные `DEC-*`: `DEC-001`
- Поля (типовой вариант):
  - `version` — `INTEGER`, primary key
  - `applied_at` — `TIMESTAMPTZ`, required, default `now()`

## Связи

- `DM-001 -> DM-002`: индексы принадлежат таблице чанков и должны соответствовать запросам store.

## Производные правила

- Порядок применения миграций фиксируется номерами/именами файлов; “ручные” операции (например смена метода индекса) описываются как playbook, а не как автоматическая миграция v1.

## Переходы состояний

- Не требуется: жизненный цикл сущностей описывается DDL+операциями Upsert/Delete.

## Вне scope

- Online reindex / CONCURRENTLY и сложные операционные процедуры миграций.
- Поддержка нескольких размерностей embeddings в одной таблице.
