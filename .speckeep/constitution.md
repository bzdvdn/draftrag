# Конституция проекта draftRAG

## Назначение

draftRAG — это Go-библиотека (пакет) для быстрого создания RAG (Retrieval-Augmented Generation) систем. Предоставляет абстракции над векторными хранилищами и LLM-провайдерами, позволяя разработчикам интегрировать RAG-функциональность в свои приложения с минимальной конфигурацией.

## Ключевые принципы

### Интерфейсная абстракция

Все внешние зависимости (LLM-провайдеры, векторные хранилища, эмбеддинг-модели) ДОЛЖНЫ быть описаны через Go-интерфейсы. Конкретные реализации ДОЛЖНЫ удовлетворять этим интерфейсам. Это позволяет заменять провайдеров без изменения клиентского кода.

### Чистая архитектура

Код ДОЛЖЕН следовать принципам Clean Architecture: разделение на слои (domain, application, infrastructure), зависимость направлена внутрь (от внешних слоёв к внутренним). Domain-слой НЕ ДОЛЖЕН импортировать внешние пакеты кроме стандартной библиотеки Go.

### Минимальная конфигурация

Пакет ДОЛЖЕН работать из коробки с разумными настройками по умолчанию. Конфигурация ДОЛЖНА быть опциональной и интуитивно понятной. Пользователь МОЖЕТ переопределить поведение через явные параметры.

### Контекстная безопасность

Все операции ДОЛЖНЫ принимать `context.Context` как первый параметр. Это обеспечивает поддержку отмены операций, таймаутов и propagation для распределённых систем.

### Тестируемость

Каждый публичный интерфейс ДОЛЖЕН иметь мок-реализацию для тестирования. Код ДОЛЖЕН быть написан так, чтобы внешние зависимости можно было заменить на тестовые заглушки.

## Ограничения

- Язык реализации: Go (версия 1.21+)
- Поддержка только Go-экосистемы (нет bindings для других языков)
- Нет встроенного HTTP-сервера — интеграция с сервером остаётся на пользователе пакета
- Нет CLI — пакет используется только как библиотека

## Технологический стек

- Язык: Go 1.21+
- Тестирование: стандартный пакет `testing` + `testify` для assert/require
- Векторные хранилища: абстракция через интерфейс `VectorStore`, реализации для PostgreSQL+pgvector, Qdrant, ChromaDB, in-memory (для тестов)
- LLM-провайдеры: абстракция через интерфейс `LLMProvider`, реализации для OpenAI-совместимых API, Anthropic, Ollama
- Эмбеддинги: абстракция через интерфейс `Embedder`, реализации для OpenAI, HuggingFace, локальные модели
- Зависимости: минимальный набор, только необходимые внешние пакеты

## Основная архитектура

### Слои

1. **Domain** (`internal/domain/`): интерфейсы `VectorStore`, `LLMProvider`, `Embedder`, domain-модели (`Document`, `Chunk`, `Query`, `RetrievalResult`)
2. **Application** (`internal/application/`): use-cases (`IndexDocument`, `RetrieveContext`, `GenerateAnswer`), оркестрация между компонентами
3. **Infrastructure** (`internal/infrastructure/`): конкретные реализации интерфейсов (pgvector, qdrant, openai, anthropic, ollama)
4. **API** (`pkg/draftrag/`): публичный API пакета, композиция компонентов, фабрики

### Поток данных

```
Document → [Embedder] → Embedding → [VectorStore] → indexed
Query → [Embedder] → Embedding → [VectorStore] → RetrievalResult → [LLMProvider] → Answer
```

### Структура каталогов

```
pkg/draftrag/        # Публичный API
internal/
  domain/            # Интерфейсы и domain-модели
  application/       # Use-cases
  infrastructure/    # Реализации внешних зависимостей
    vectorstore/     # pgvector, qdrant, chromadb, memory
    llm/             # openai, anthropic, ollama
    embedder/        # openai, huggingface, local
```

## Приоритеты решений

- **Простота против расширяемости**: предпочитать простоту, если расширяемость не требуется для поддержки множественных провайдеров
- **Корректность против скорости**: корректность и безопасность важнее микросекундных оптимизаций
- **Поддерживаемость против cleverness**: избегать метапрограммирования и сложных паттернов без явной необходимости
- **Интерфейсы против конкретных типов**: предпочитать интерфейсы для внешних зависимостей, конкретные типы для internal domain-логики

## Ключевые измерения качества

- **Тестовое покрытие**: ≥80% для domain и application слоёв, ≥60% для infrastructure
- **Время сборки**: `go build ./...` ДОЛЖЕН завершаться <5 секунд на среднем ноутбуке
- **Публичный API стабильность**: семантическое версионирование, breaking changes только в major-версиях
- **Документация**: каждый публичный тип и функция ДОЛЖНЫ иметь godoc-комментарий на русском языке

## Языковая политика

- Язык документации: русский
- Язык общения с агентом: русский
- Язык комментариев в коде: русский
- Godoc-комментарии: русский
- Имена переменных и функций: английский (стандарт Go)

## Процесс разработки

- Каждая фича ДОЛЖНА разрабатываться в отдельной git-ветке с префиксом `feature/<slug>`
- Реализация ДОЛЖНА начинаться с явной спецификации в `.draftspec/specs/<slug>/spec.md`
- План и задачи ДОЛЖНЫ выводиться из спецификации и размещаться в `.draftspec/plans/<slug>/`
- Код ДОЛЖЕН проходить `go vet`, `go fmt`, `golangci-lint` без ошибок и предупреждений
- Все новые функции ДОЛЖНЫ иметь unit-тесты
- Реализация, спецификации, планы и задачи ДОЛЖНЫ соответствовать этой конституции
- Если работа выявляет конфликт с этой конституцией, конституция ДОЛЖНА быть изменена до продолжения несовместимой реализации
- Spec, inspect, plan, tasks или implement НЕ ДОЛЖНЫ продолжаться, если они нарушают принципы Clean Architecture или интерфейсную абстракцию

## Управление

- Эта конституция является авторитетным источником для проектных решений
- Изменения архитектуры, спецификаций, планов и задач ДОЛЖНЫ соответствовать этим принципам
- Если реализация конфликтует с конституцией, приоритет у конституции, пока она явно не изменена
- Изменяйте этот файл patch-обновлениями, сохраняя обязательные секции и делая правила конкретными и проверяемыми
- `.draftspec/constitution.summary.md` ДОЛЖЕН обновляться синхронно с основной конституцией

## Протокол исключений

- Отклонения от конституции ДОЛЖНЫ быть явно задокументированы в commit-сообщении и в соответствующей спецификации
- Breaking changes в публичном API ДОЛЖНЫ требовать обновления major-версии и миграционного гайда
- Временные отступления (technical debt) ДОЛЖНЫ быть помечены `TODO`-комментариями с ссылкой на issue
- Downstream фазы (spec, plan, implement) ДОЛЖНЫ считать конфликт с конституцией BLOCKER, если нет явного исключения, задокументированного в spec
