# Production checklist + runbook

## Scope Snapshot

- In scope: добавить “production-ready” документ (checklist + runbook) для эксплуатации draftRAG: таймауты, ретраи/CB, миграции pgvector, коллекции Qdrant/Weaviate, observability (hooks/OTel), безопасное логирование (redaction) и типовые инциденты.
- Out of scope: изменение публичного API, добавление новых бэкендов, полноценные SLO/дашборды, автоматическая диагностика/самовосстановление, инфраструктурные шаблоны деплоя (K8s charts/terraform).

## Цель

Команды (стартапы и энтерпрайз) должны получить один короткий и практичный документ, который снижает риск ошибок в проде: что проверить перед релизом, как “правильно” включать таймауты/ретраи/кеш, как безопасно запускать миграции и что делать при типовых сбоях (пустая выдача, рост latency/стоимости, circuit breaker open, ошибки миграций).

## Основной сценарий

1. Пользователь готовит сервис с draftRAG к продакшену.
2. Пользователь открывает “Production checklist + runbook”.
3. Пользователь проходит чеклист (конфигурация, миграции, лимиты, логирование/редакция, наблюдаемость).
4. При инциденте пользователь открывает runbook и следует шагам диагностики (по метрикам/логам/хукам) и mitigation.

## Scope

- Создать новый docs-артефакт “Production checklist + runbook” (в `docs/`) и добавить ссылку на него из `README.md`.
- Документ ДОЛЖЕН покрывать эксплуатационные темы, которые уже присутствуют в проекте:
  - таймауты и `context.Context`;
  - retry/circuit breaker;
  - кэш эмбеддингов (L1/L2);
  - pgvector миграции и runtime timeouts;
  - коллекции Qdrant/Weaviate;
  - observability hooks и OTel hooks;
  - redaction секретов и границы ответственности.
- Документ должен быть кратким и ориентированным на действия (checklist + runbook шаги), без “эссе”.

## Контекст

- README уже содержит production-примеры и отдельные секции по retry/circuit breaker, hooks/OTel и redaction; но нет единого “операционного” entrypoint.
- По конституции проект ориентирован на контекстную безопасность, минимальную конфигурацию и production пригодность.
- Проект содержит разные backend’ы; документ должен фиксировать различия, влияющие на эксплуатацию (миграции/коллекции/таймауты).

## Требования

- RQ-001 ДОЛЖЕН существовать единый документ “Production checklist + runbook” в `docs/`, на русском языке.
- RQ-002 Документ ДОЛЖЕН содержать checklist перед релизом (5–15 пунктов) с конкретными, проверяемыми действиями.
- RQ-003 Документ ДОЛЖЕН содержать runbook для минимум 4 типовых инцидентов (симптомы → диагностика → действия).
- RQ-004 Документ ДОЛЖЕН ссылаться на существующие публичные API/настройки (timeouts, retry/CB, cache, hooks/OTel, migrations) без введения новых требований к коду.
- RQ-005 В `README.md` ДОЛЖНА быть ссылка на этот документ в разделе документации/примеров.

## Вне scope

- Изменения кода библиотеки ради документа (кроме минимальных правок README для ссылки).
- Введение новых метрик/хуков/редакции — это отдельные фичи.
- Генерация готовых конфигов (Kubernetes, docker-compose) и шаблонов деплоя.

## Критерии приемки

### AC-001 Документ checklist+runbook добавлен и доступен из README

- Почему это важно: единая точка входа снижает время онбординга и риск “пропустить важное”.
- **Given** пользователь читает `README.md`
- **When** он ищет “как запускать в production”
- **Then** он находит ссылку на документ и может открыть его в `docs/`
- Evidence: в `README.md` есть явная ссылка, и файл документа присутствует в репозитории

### AC-002 Checklist содержит конкретные действия по конфигурации и миграциям

- Почему это важно: больше всего инцидентов — из-за таймаутов/миграций/лимитов и неверного wiring.
- **Given** пользователь готовит релиз
- **When** он следует checklist
- **Then** каждый пункт содержит проверяемое действие (например, “миграции pgvector запускаются отдельной job”, “таймауты заданы через context”, “ретраи включены с лимитами”, “кеш настроен”) без двусмысленности
- Evidence: checklist содержит 5–15 пунктов, каждый — измерим/проверяем

### AC-003 Runbook покрывает типовые инциденты и шаги диагностики

- Почему это важно: в инциденте нужен быстрый сценарий действий, а не поиск по README.
- **Given** сервис в проде деградирует
- **When** возникает один из типовых инцидентов (пустая выдача, рост latency, circuit breaker open, ошибки миграций/коллекций)
- **Then** документ содержит: симптомы, что проверить (метрики/логи/хуки), и конкретные mitigation шаги
- Evidence: минимум 4 раздела runbook с одинаковой структурой “Symptoms/Checks/Actions”

### AC-004 Документ фиксирует границы ответственности и безопасность логов

- Почему это важно: enterprise комплаенс и снижение риска утечек.
- **Given** пользователь включает observability/logging
- **When** он читает раздел про безопасность
- **Then** он видит, что библиотека редактирует известные секреты best-effort, но пользователь отвечает за свои логи и контент документов/запросов
- Evidence: в документе есть отдельный короткий подраздел про redaction и границы ответственности

## Допущения

- Документ не должен требовать изменений кода, кроме ссылки из README.
- “Production checklist” — это руководство, а не гарантия SLO; оно задаёт безопасные стартовые практики.

## Краевые случаи

- Разные store backend’ы: document должен явно отметить, где нужны миграции (pgvector) и где нужны операции с коллекциями (Qdrant/Weaviate).
- High-cardinality observability: упомянуть риск кардинальности метрик/labels и как его ограничивать (на уровне рекомендаций).
