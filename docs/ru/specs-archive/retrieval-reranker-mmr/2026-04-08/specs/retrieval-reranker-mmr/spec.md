# Retrieval: MMR rerank (диверсификация источников) (v1)

## Scope Snapshot

- In scope: добавить опциональный rerank/selection шаг MMR (Maximal Marginal Relevance) для повышения разнообразия контекста при сохранении релевантности.
- Out of scope: cross-encoder reranker, LLM-based reranking, обучение моделей.

## Контекст

- Сейчас topK retrieval часто приносит очень похожие чанки (особенно после чанкинга с overlap), из-за чего prompt “засоряется” повторяющимися фрагментами.
- Это ухудшает качество ответа: LLM видит меньше разнообразных фактов/аспектов вопроса при том же лимите контекста.
- Нужен лёгкий и детерминированный rerank/selection шаг без внешних моделей: использовать embeddings, которые уже доступны в `Chunk`.

## Цель

Снизить “overlap” источников: вместо 5 почти одинаковых чанков дать 3–5 разных, покрывающих разные аспекты вопроса.

## Основной сценарий

1. Retrieval возвращает кандидатов (например topK=20).
2. Pipeline применяет MMR, выбирая N чанков с балансом релевантность/разнообразие.
3. В prompt попадает более полезный контекст.

## Требования

- RQ-001 MMR ДОЛЖЕН быть опциональным (по умолчанию выключен).
- RQ-002 ДОЛЖНА быть конфигурация `Lambda` (0..1) и `CandidatePool` (сколько кандидатов брать до отбора).
- RQ-003 MMR ДОЛЖЕН работать без внешних моделей: использовать уже имеющиеся embeddings чанков и query embedding.
- RQ-004 Контекстная отмена ДОЛЖНА сохраняться (ctx cancellation).
- RQ-005 Тесты ДОЛЖНЫ быть детерминированными и без внешней сети.

## Критерии приемки

### AC-001 MMR меняет выбор источников при наличии дубликатов

- **Given** есть несколько почти одинаковых чанков с высокими score
- **When** включён MMR
- **Then** итоговый набор содержит более разнообразные чанки (не все из одного “кластера”)

### AC-002 Выключенный MMR не влияет на поведение

- **Given** MMR выключен
- **When** выполняется Answer
- **Then** поведение совпадает с текущим (только sort by score)
