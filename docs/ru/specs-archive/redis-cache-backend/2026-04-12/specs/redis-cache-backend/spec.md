# Redis backend для кэша эмбеддингов

## Scope Snapshot

- In scope: добавить опциональный Redis (L2) как backend для кэша эмбеддингов поверх существующего in-memory LRU (L1), чтобы кэш работал при горизонтальном масштабировании.
- Out of scope: кэширование для других компонентов (retriever/vector store/LLM), распределённые блокировки, дедупликация конкурентных запросов (singleflight) между инстансами.

## Цель

Пользователь `CachedEmbedder` может включить Redis как second-level cache, чтобы повторные эмбеддинги для одного и того же текста не уходили в внешний embedder даже при нескольких экземплярах приложения. Успех виден по тому, что при L1 miss и L2 hit запрос не вызывает базовый embedder и результат кладётся обратно в L1.

## Основной сценарий

1. Пользователь создаёт `CachedEmbedder` и (опционально) передаёт настройки Redis (клиент, TTL, префикс ключей).
2. На `Embed(ctx, text)` вычисляется стабильный ключ (на базе SHA-256 от `text`).
3. Сначала проверяется L1 (in-memory LRU); при попадании возвращается значение.
4. При L1 miss и включённом Redis выполняется чтение из Redis; при попадании значение десериализуется, кладётся в L1 и возвращается.
5. При L1+L2 miss вызывается базовый embedder, результат сохраняется в L1 и best-effort записывается в Redis (с TTL, если задан).

## Scope

- Публичная конфигурация `CachedEmbedder` расширяется так, чтобы можно было включать Redis без breaking changes для существующего кода.
- Реализуется Redis backend для хранения векторов: сериализация/десериализация `[]float64` в `[]byte` (msgpack) и операции `Get/Set` с TTL.
- Поведение при недоступности Redis: graceful degradation (логирование/пропуск Redis) без влияния на корректность `Embed` при живом базовом embedder.

## Контекст

- Сейчас `CachedEmbedder` доступен пользователям из `pkg/draftrag` и использует только in-memory LRU, что не подходит для горизонтального масштабирования (см. `ROADMAP.md`, “Embedder cache: Redis backend”).
- В репозитории уже есть зависимость `msgpack` и внутренняя заготовка опции `WithRedis(...)` в `internal/infrastructure/embedder/cache`, но отсутствует реализованный Redis backend.
- В соответствии с конституцией, внешние зависимости должны быть абстрагированы интерфейсами; Redis должен быть опциональным и не требовать его наличия в runtime, если пользователь его не включает.

## Требования

- RQ-001 `CachedEmbedder` ДОЛЖЕН сохранять существующее поведение “только LRU” по умолчанию (без Redis), без изменений в вызывающем коде.
- RQ-002 ДОЛЖНА быть доступна опция включения Redis (L2) из публичного API `pkg/draftrag` без необходимости импортировать internal-пакеты.
- RQ-002.1 Публичный API ДОЛЖЕН принимать Redis-клиент через адаптер-интерфейс (минимальный контракт), чтобы не привязывать библиотеку к конкретной реализации клиента.
- RQ-003 При L1 miss и включённом Redis кэш ДОЛЖЕН пытаться прочитать значение из Redis и, при успехе, вернуть его без вызова базового embedder.
- RQ-004 При L2 hit значение ДОЛЖНО быть записано в L1 (warming), чтобы последующие обращения на этом инстансе обслуживались из L1 и не выполняли `Redis GET`.
- RQ-005 При L1+L2 miss кэш ДОЛЖЕН вызывать базовый embedder и сохранять результат в L1; запись в Redis ДОЛЖНА быть best-effort и не ломать `Embed`, если Redis не доступен.
- RQ-006 В Redis ДОЛЖНЫ храниться сериализованные векторы (`[]float64`) в виде байтов; формат сериализации должен быть стабильным внутри версии библиотеки (msgpack).
- RQ-007 Опция Redis ДОЛЖНА поддерживать TTL (0 → без TTL) и префикс ключей (чтобы избегать коллизий между приложениями/окружениями); если префикс не задан, используется дефолт `draftrag:embedder:`.
- RQ-008 Любая ошибка чтения/десериализации данных из Redis ДОЛЖНА рассматриваться как cache miss (treat-as-miss) с логированием и не должна ломать `Embed`.
- RQ-009 Новая функциональность ДОЛЖНА быть покрыта unit-тестами без необходимости поднимать реальный Redis.

## Вне scope

- Общий интерфейс кэша для произвольных типов, универсальный “CacheBackend” для других частей библиотеки.
- Интеграционные тесты, требующие запущенного Redis (docker-compose и т.п.).
- Механизмы борьбы с “stampede” (singleflight) и распределённая координация между инстансами.
- Политики eviction/лимиты памяти в Redis, мониторинг Redis, метрики по слоям L1/L2.

## Критерии приемки

### AC-001 Публичная опция Redis не ломает текущих пользователей

- Почему это важно: включение Redis — опциональное улучшение; текущие интеграции не должны требовать изменений.
- **Given** пользовательский код создаёт `CachedEmbedder` без параметров Redis
- **When** выполняются повторные вызовы `Embed(ctx, sameText)`
- **Then** поведение остаётся как сейчас: LRU-кэширует и не требует наличия Redis-клиента
- Evidence: существующие примеры/тесты компилируются и проходят без дополнительной конфигурации Redis.

### AC-002 L2 hit не вызывает базовый embedder

- Почему это важно: основная ценность Redis — кэширование между инстансами и снижение стоимости/латентности.
- **Given** L1 пуст, Redis включён и содержит валидное значение для ключа текста
- **When** вызывается `Embed(ctx, text)`
- **Then** возвращается вектор из Redis и базовый embedder не вызывается
- Evidence: unit-тест с mock embedder (счётчик вызовов) и fake Redis, где `embedder.Embed` остаётся 0.

### AC-003 Warming L1 после L2 hit

- Почему это важно: после первого L2 hit последующие обращения должны обслуживаться локально (без сети/Redis).
- **Given** L1 пуст, Redis включён и содержит значение для ключа
- **When** `Embed(ctx, text)` вызывается два раза подряд
- **Then** второй вызов обслуживается из L1 без обращения к Redis
- Evidence: unit-тест с подсчётом вызовов fake Redis `Get`.

### AC-004 Best-effort запись в Redis и graceful degradation

- Почему это важно: Redis может быть временно недоступен; библиотека должна продолжать работать.
- **Given** Redis включён, но операции `Get/Set` возвращают ошибку
- **When** вызывается `Embed(ctx, text)`
- **Then** возвращается результат базового embedder, а ошибка Redis не “пробивает” наружу как ошибка `Embed`
- Evidence: unit-тест, где fake Redis всегда возвращает ошибку, но `Embed` успешен при успешном embedder.

### AC-005 TTL и префикс ключей учитываются

- Почему это важно: TTL контролирует stale-данные, префикс избегает коллизий в shared Redis.
- **Given** Redis включён с TTL и префиксом ключей
- **When** выполняется запись в Redis после miss
- **Then** Redis `Set` вызывается с TTL и ключом, начинающимся с префикса
- Evidence: unit-тест, который проверяет аргументы вызова fake Redis `Set`.

### AC-006 Повреждённые данные в Redis не ломают Embed

- Почему это важно: в shared Redis могут оказаться некорректные/устаревшие значения; библиотека должна быть устойчивой.
- **Given** Redis включён и возвращает байты, которые не десериализуются в `[]float64`
- **When** вызывается `Embed(ctx, text)`
- **Then** поведение эквивалентно miss: вызывается базовый embedder и результат сохраняется в кэш
- Evidence: unit-тест с fake Redis, возвращающим “битые” байты.

## Допущения

- Пользователь предоставляет Redis-клиент, который потокобезопасен и корректно использует `context.Context`.
- Вектор эмбеддинга представлен как `[]float64` и остаётся сериализуемым/десериализуемым без потери смысла для сценариев использования библиотеки.
- Стабильность ключа: SHA-256 от текста (или иной неизменный алгоритм) достаточно для практического устранения коллизий.

## Критерии успеха

- SC-001 При включённом Redis и повторных запросах между инстансами доля вызовов базового embedder снижается (измеряется в приложении пользователя через счётчики вызовов/логирование; библиотека не обязана включать метрики).

## Краевые случаи

- TTL = 0 (без TTL): значения в Redis остаются до явного удаления/eviction в Redis.
- Redis возвращает значение, размер которого слишком большой/невалидный → обработка как miss.
- Redis недоступен/таймаут → `Embed` продолжает работать через базовый embedder.
