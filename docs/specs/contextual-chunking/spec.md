# Contextual Chunking

## Scope Snapshot

- In scope: добавить `ContextualChunker` — декоратор над любым `Chunker`, который обогащает каждый чанк документным контекстом (например, заголовок документа) перед возвратом.
- Out of scope: динамическое извлечение section headings, multi-level контекст, LLM-based контекст, изменение формата хранения чанков в VectorStore.

## Цель

Разработчик, использующий draftRAG, получает возможность индексировать документы так, чтобы каждый чанк содержал релевантный контекст документа (заголовок, краткое описание). Это повышает качество retrieval: при поиске чанк интерпретируем отдельно от исходного документа. Успех фичи измеряется — контекстный префикс присутствует в `Chunk.Content` и участвует в эмбеддинге.

## Основной сценарий

1. Пользователь создаёт `ContextualChunker`, передавая базовый `Chunker` и опции (контекстный шаблон, источник контекста).
2. `ContextualChunker.Chunk(ctx, doc)` сначала вызывает базовый `Chunker.Chunk(ctx, doc)`.
3. Для каждого полученного чанка `ContextualChunker` формирует контекстную строку из полей документа (например, `doc.Metadata["title"]`).
4. Контекстная строка подставляется в заданный шаблон (по умолчанию `"[CONTEXT] {context}\n{content}"`) и становится новым `Chunk.Content`.
5. Результирующие чанки возвращаются вызывающему коду как обычно.
6. Если источник контекста пуст (`""`) или отсутствует, чанки возвращаются без изменений.

## User Stories

- P1 Story: как разработчик RAG-пайплайна, я хочу добавить заголовок документа в каждый чанк, чтобы при поиске чанк был понятен без исходного документа.
- P1 Story: как пользователь `Pipeline`, я хочу указать, из какого поля `Document.Metadata` брать контекст, чтобы гибко настраивать источник.

## MVP Slice

Наименьший срез: `ContextualChunker`, который принимает базовый `Chunker`, имя поля метаданных (по умолчанию `"title"`) и шаблон префикса. Закрывает AC-001, AC-002, AC-003, AC-004, AC-006.

## First Deployable Outcome

Можно написать тест: создать `ContextualChunker` с `BasicChunker`, вызвать `Chunk` на документе с `Metadata["title"]="Отчёт"`, проверить что каждый чанк начинается с `"[CONTEXT] Отчёт\n"` и содержит исходный контент.

## Scope

- Новый `ContextualChunker` — декоратор над `domain.Chunker`, который модифицирует `Chunk.Content` после вызова внутреннего чанкера.
- Настраиваемый шаблон контекста (строка с плейсхолдерами `{context}` и `{content}`).
- Настраиваемый источник контекста: поле из `Document.Metadata`.
- Контекст добавляется **перед** тем, как чанк будет передан в эмбеддер и сохранён.
- Валидация опций (пустой шаблон, неизвестный плейсхолдер) на уровне конструктора.
- Context cancellation и пустой документ обрабатываются как в базовых чанкерах.

## Контекст

- Существующий интерфейс `domain.Chunker` не меняется — `ContextualChunker` реализует тот же интерфейс через композицию.
- `Document.Metadata` уже существует и может содержать любые строковые пары ключ-значение.
- `Chunk` не имеет отдельного поля для контекста — контекст становится частью `Content`.
- Никакие другие части пайплайна (эмбеддеры, VectorStore, retrieval) не требуют изменений.

## Зависимости

- `none` — фича не вводит внешних зависимостей. Использует только существующие `domain.Document`, `domain.Chunk`, `domain.Chunker`.

## Требования

- RQ-001 Система ДОЛЖНА предоставлять `ContextualChunker`, реализующий `domain.Chunker` через композицию с другим `Chunker`.
- RQ-002 `ContextualChunker.Chunk` ДОЛЖНА извлекать контекст из `Document.Metadata` по заданному ключу.
- RQ-003 `ContextualChunker.Chunk` ДОЛЖНА подставлять контекст в шаблон вместе с содержимым чанка.
- RQ-004 Если контекст пуст или ключ не найден в `Document.Metadata`, чанки ДОЛЖНЫ возвращаться без изменений.
- RQ-005 Валидация опций ДОЛЖНА отклонять пустой шаблон и шаблон без `{content}`.

## Вне scope

- Изменение `Chunk` или `Document` моделей (новые поля).
- Автоматическое извлечение контекста (заголовков, секций) из текста документа — пользователь указывает контекст явно через `Document.Metadata`.
- Поддержка множественных источников контекста (несколько полей метаданных) — только одно поле.
- LLM-генерация контекста/саммари документа.
- Изменение формата хранения в VectorStore.

## Критерии приемки

### AC-001 Контекст добавляется к чанкам

- Почему это важно: основной сценарий использования — контекст должен быть добавлен.
- **Given** документ с `Metadata["title"]="Research Paper"` и базовый `BasicChunker`
- **When** вызван `ContextualChunker.Chunk(ctx, doc)`
- **Then** каждый чанк содержит префикс с заголовком документа
- Evidence: `strings.HasPrefix(chunk.Content, "[CONTEXT] Research Paper\\n")` для всех возвращённых чанков

### AC-002 Настраиваемый шаблон контекста

- Почему это важно: пользователь может захотеть другой формат контекста.
- **Given** `ContextualChunker` с кастомным шаблоном `"Doc: {context} --- {content}"`
- **When** вызван `Chunk(ctx, doc)` с документом, имеющим контекст
- **Then** каждый чанк использует заданный шаблон
- Evidence: `strings.HasPrefix(chunk.Content, "Doc: ")` и контент следует после `---`

### AC-003 Пустой контекст не модифицирует чанки

- Почему это важно: документы без контекста не должны ломаться.
- **Given** документ без `Metadata["title"]` (или с пустым значением)
- **When** вызван `ContextualChunker.Chunk(ctx, doc)`
- **Then** каждый чанк возвращается без изменений (как от базового чанкера)
- Evidence: содержимое чанков идентично вызову базового `Chunker.Chunk(ctx, doc)` без контекстного декоратора

### AC-004 Уважение context cancellation

- Почему это важно: консистентность с архитектурным требованием `context.Context` во всех публичных операциях.
- **Given** отменённый контекст
- **When** вызван `ContextualChunker.Chunk(ctx, doc)`
- **Then** возвращается ошибка `ctx.Err()`
- Evidence: вызов с отменённым контекстом возвращает ошибку, не выполняя чанкинг

### AC-005 Контекст влияет на эмбеддинг

- Почему это важно: контекст — не просто метаданные, он часть поискового представления.
- **Given** документ с контекстом, проиндексированный через Pipeline
- **When** выполнен поиск по запросу, совпадающему с контекстом
- **Then** соответствующий чанк возвращается в результатах
- Evidence: поиск по слову из контекста (которого нет в исходном тексте чанка) возвращает этот чанк

### AC-006 Настраиваемый источник контекста из метаданных

- Почему это важно: пользователь может хранить контекст в любом поле Metadata.
- **Given** `ContextualChunker` с `ContextKey="description"` и документ с `Metadata["description"]="Annual Report 2025"`
- **When** вызван `Chunk(ctx, doc)`
- **Then** контекст берётся из `Metadata["description"]`
- Evidence: чанки содержат префикс с "Annual Report 2025"

## Допущения

- Контекст — это статическая строка, известная на момент индексации (берётся из `Document.Metadata`).
- Шаблон содержит ровно `{context}` и `{content}` плейсхолдеры. Другие плейсхолдеры не поддерживаются.
- `ContextualChunker` не кеширует результаты базового чанкера — каждый вызов `Chunk` выполняет полный цикл.

## Критерии успеха

- SC-001 Все acceptance criteria покрыты unit-тестами.
- SC-002 Время выполнения `Chunk` для `ContextualChunker` не превышает время базового чанкера + O(n) на конкатенацию строк (где n — число чанков).

## Краевые случаи

- Документ с пустым `Content` — возвращается пустой срез.
- Документ с `Metadata` равным `nil` — обрабатывается как отсутствие контекста (AC-003).
- Шаблон без плейсхолдера `{content}` — ошибка валидации.
- Пустой шаблон — ошибка валидации.
- Очень длинный контекст (например, 10k символов) — используется как есть, без обрезки.
