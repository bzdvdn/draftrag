# Retrieval: дедупликация источников (v1) — Модель данных

## Scope

- Связанные `AC-*`: `AC-001`, `AC-002`
- Связанные `DEC-*`: `DEC-001`, `DEC-002`
- Persisted data model не меняется; меняется только форма retrieval результата перед использованием в prompt/выдаче.

## Сущности

### DM-001 DedupConfig (настройка дедупликации)

- Назначение: включать/выключать дедупликацию и выбирать режим.
- Источник истины:
  - `pkg/draftrag.PipelineOptions` (публичная опция),
  - `internal/application.PipelineConfig` (прокинутый config).
- Инварианты:
  - по умолчанию дедупликация выключена.
  - v1 поддерживает только дедуп по `ParentID`.
- Связанные `AC-*`: `AC-002`
- Связанные `DEC-*`: `DEC-002`
- Поля:
  - `Enabled` — `bool`
  - `Mode` — enum-like (v1: `ParentID`)

### DM-002 DedupedRetrievalResult (результат после дедупликации)

- Назначение: уменьшить повторы в `RetrievalResult.Chunks` перед построением prompt и/или возвратом пользователю.
- Источник истины: функция дедупликации в application слое.
- Инварианты:
  - при включённой дедупликации: не более одного чанка на `ParentID`.
  - выбирается лучший по `Score` (max).
  - при равных score выбор детерминированный (stable, первый встретившийся).
- Связанные `AC-*`: `AC-001`, `AC-002`
- Поля:
  - `Chunks` — `[]domain.RetrievedChunk`
  - `QueryText` — не меняется
  - `TotalFound` — не меняется (это счётчик до topK; v1 не меняем семантику)

## Связи

- `DM-001` управляет применением к `DM-002` в retrieval path.

## Производные правила

- Дедупликация применяется после сортировки по релевантности (или эквивалентного порядка от store) и не должна “перемешивать” элементы без необходимости.

## Переходы состояний

- Не применимо.

## Вне scope

- Дедупликация по контентному hash.
- Семантическая дедупликация через отдельные модели.
