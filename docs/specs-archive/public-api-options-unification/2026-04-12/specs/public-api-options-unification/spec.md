# Единый options pattern в публичном API

## Scope Snapshot

- In scope: унифицировать публичные конструкторы `pkg/draftrag` на единый паттерн options-конфигурации, чтобы пользовательский опыт был консистентным (без “угадывания”, где struct, где functional options).
- Out of scope: изменение внутренних (internal) паттернов конфигурации, тотальный рефакторинг всех типов/имен, а также добавление новых фич-провайдеров.

## Цель

Сделать публичный API draftRAG предсказуемым: любой “конфигурируемый” конструктор использует один и тот же подход к опциям, а все существующие вызовы остаются совместимыми (либо получают понятный миграционный путь). Успех виден по тому, что пользователь может применять одинаковый стиль конфигурации для embedder’ов, LLM’ов, vector store’ов, кэша и resilience-обёрток.

## Основной сценарий

1. Пользователь выбирает компонент (например, `OpenAICompatibleLLM`, `QdrantStore`, `CachedEmbedder`).
2. Пользователь настраивает компонент через единый паттерн options (одинаковый для всех конструкторов).
3. Документация и примеры используют тот же паттерн, без исключений.
4. Старый стиль вызова (если был) продолжает работать в рамках backward compatibility или имеет явную депрекацию/миграцию.

## Scope

- Определить и зафиксировать один публичный options-паттерн для `pkg/draftrag` (выбор между “struct Options” и “functional options”).
- Привести к нему основные публичные конструкторы:
  - Embedders: `NewOpenAICompatibleEmbedder`, `NewOllamaEmbedder`, …
  - LLM: `NewOpenAICompatibleLLM`, `NewAnthropicLLM`, `NewOllamaLLM`, …
  - Vector stores: `NewQdrantStore`, `NewPGVectorStore`, `NewWeaviateStore`, `NewChromaDBStore`, …
  - Wrappers: `NewCachedEmbedder`, `NewRetryEmbedder`, `NewRetryLLMProvider`, `NewPipelineWithOptions`
- Обновить документацию и примеры, чтобы они использовали единый паттерн.
- Добавить tests/compile checks для ключевых путей, чтобы предотвратить регрессию консистентности.

## Контекст

- Сейчас публичный API в основном использует `...Options` struct, но в некоторых местах есть internal functional options (например, кэш), а также разные стили “runtime options” (pgvector).
- Для пользователей библиотек консистентность API важнее, чем внутренняя реализация: единый подход снижает когнитивную нагрузку и ускоряет onboarding.
- Конституция требует “минимальной конфигурации”: важно сохранить дефолты и простые вызовы без явной конфигурации.

## Требования

- RQ-001 ДОЛЖЕН быть выбран один публичный options-паттерн для `pkg/draftrag` и применён консистентно.
- RQ-002 Изменения НЕ ДОЛЖНЫ ломать существующий код без явной миграционной истории (или должны быть оформлены как major/breaking с гайдом — если это осознанное решение).
- RQ-003 Дефолтное поведение компонентов ДОЛЖНО сохраниться при отсутствии опций.
- RQ-004 Документация (README/docs/examples) ДОЛЖНА использовать единый паттерн.
- RQ-005 ДОЛЖНЫ быть добавлены проверки/тесты, которые не дадут “съехать” консистентности (хотя бы на уровне lint/rg/compile).

## Вне scope

- Переписывание internal слоёв на те же паттерны (это может идти отдельной фичей при необходимости).
- Добавление новых провайдеров, хранилищ или алгоритмов.
- Полная стандартизация имён (`Options` vs `Config`) во всём репозитории вне публичного API.

## Критерии приемки

### AC-001 Выбран и описан единый паттерн options

- Почему это важно: без чёткого правила консистентность быстро деградирует.
- **Given** reviewer смотрит на публичные конструкторы `pkg/draftrag`
- **When** он сравнивает сигнатуры и примеры использования
- **Then** видит один и тот же паттерн конфигурации и явное описание правила в документации
- Evidence: обновлённый `CONTRIBUTING.md`/`README.md` с правилом и примерами.

### AC-002 Публичные конструкторы приведены к паттерну без регрессий

- Почему это важно: пользователи ожидают одинаковый UX во всех компонентах.
- **Given** набор основных конструкторов (`embedder`, `llm`, `store`, `wrappers`)
- **When** пользователь конфигурирует любой из них
- **Then** он использует одинаковый стиль options и получает те же дефолты, что и раньше
- Evidence: `go test ./...` проходит; ключевые примеры/сниппеты компилируются.

### AC-003 Backward compatibility или миграция явны

- Почему это важно: библиотека должна быть безопасной для обновления.
- **Given** существующий пользовательский код на текущей версии
- **When** он обновляет библиотеку
- **Then** либо всё продолжает работать, либо есть чёткий миграционный раздел с заменами вызовов
- Evidence: документированная миграция + (по возможности) compile-time checks/тесты.

### AC-004 Документация и примеры унифицированы

- Почему это важно: документация — основная точка входа для пользователей.
- **Given** README, docs и examples
- **When** пользователь копирует пример
- **Then** он всегда видит единый паттерн options
- Evidence: обновлённые `README.md`, `docs/*`, `examples/*`.

### AC-005 Добавлен guardrail против дрейфа

- Почему это важно: консистентность “съезжает” со временем, если её не проверять.
- **Given** новый PR добавляет/меняет публичный конструктор
- **When** CI/проверки запускаются
- **Then** они сигналят о нарушении паттерна options
- Evidence: простой test/lint/rg check, закрепляющий правило.

## Допущения

- Публичный API может быть расширен без breaking changes путём добавления overload/новых конструкторов или дополнительных опций.
- Пользователи предпочитают предсказуемость API даже ценой небольшого verbosity.

## Критерии успеха

- SC-001 Количество “особых случаев” в публичном API снижается (все основные конструкторы следуют одному паттерну).

## Краевые случаи

- Компоненты с двумя наборами опций (например, “runtime options”) должны иметь ясное правило (либо вложенные options, либо отдельный конструктор), но в том же паттерне.

## Открытые вопросы

- Какой именно паттерн выбираем как canonical: `Options struct` (как сейчас) или functional options? (это решение должно быть принято на фазе plan).
