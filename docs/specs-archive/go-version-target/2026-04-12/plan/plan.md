# go-version-target — План

## Phase Contract

Inputs: `.speckeep/specs/go-version-target/spec.md`, `.speckeep/specs/go-version-target/inspect.md`, текущее состояние `go.mod` и docs.  
Outputs: `plan.md`, `data-model.md`.  
Stop if: выбранная “минимальная версия Go” конфликтует с фактическим кодом/зависимостями (невозможно собрать/протестировать без поднятия минимума).

## Цель

Выбрать и зафиксировать минимально поддерживаемую версию Go для draftRAG и выровнять `go.mod`, документацию и автоматические проверки так, чтобы пользователи не сталкивались с неожиданным требованием скачивать более новый toolchain.

## Scope

- Принять DEC по минимальной версии Go (канонический минимум).
- Выровнять:
  - `go.mod` (директива `go ...`, и только при необходимости — `toolchain`),
  - `.speckeep/constitution.md` (если потребуется изменить минимум),
  - `README.md` и `docs/getting-started.md`.
- Добавить минимальный CI (GitHub Actions), прогоняющий `go test ./...` на минимальной версии.

## Implementation Surfaces

- `go.mod`
- `.speckeep/constitution.md` (возможно, без изменений)
- `README.md`
- `docs/getting-started.md`
- `.github/workflows/ci.yml` (новый)

## Решения (DEC)

### DEC-001 Минимально поддерживаемая версия Go: 1.23

Why:
- Зависимости проекта (например, `github.com/jackc/pgx/v5`) требуют Go 1.23+.
- Фактический минимум уже отражён в `go.mod` и документации; нужно выровнять конституцию и CI под реальность.

Tradeoff:
- Пользователям потребуется Go 1.23+ (это повышает “порог входа”, но соответствует фактической совместимости).

Validation:
- `go.mod` имеет `go 1.23.x`.
- `go test ./...` проходит на Go 1.23 в CI.

### DEC-002 Директива `toolchain` в `go.mod`: допускается, но по умолчанию не используем

Why:
- `toolchain go1.xx` может провоцировать автоскачивание toolchain у downstream пользователей, что противоречит UX “минимальная версия = достаточно”.
- Dev-удобство можно получать через локально установленный Go или через CI matrix, не навязывая toolchain в модуле.

Tradeoff:
- Разработчики не будут автоматически “прибиты” к версии toolchain через `go.mod`.

Validation:
- В `go.mod` отсутствует `toolchain` (или она равна минимальной версии и не повышает требования).
- Документация чётко говорит “минимальная версия Go: 1.23”.

### DEC-003 CI: проверяем минимум через GitHub Actions matrix

Why:
- Без автоматики версия быстро “поплывёт”.

Tradeoff:
- Добавление `.github/workflows/ci.yml` увеличит инфраструктуру репозитория, но даёт явный guardrail.

Validation:
- Workflow запускает `go test ./...` как минимум на Go 1.23.
- Дополнительно можно запускать на latest stable, но минимум обязателен.

## Acceptance Approach

- AC-001: изменение `go.mod` + подтверждение тестами на минимальной версии.
- AC-002: обновление `README.md` и `docs/getting-started.md`, при необходимости — `.speckeep/constitution.md`.
- AC-003: добавление `.github/workflows/ci.yml` с `actions/setup-go` и `go test ./...` на Go 1.23.

## Порядок реализации

1. Принять DEC-001/DEC-002 и обновить docs с единым минимумом.
2. Обновить `go.mod` на `go 1.23.x` (и убрать/не добавлять `toolchain`).
3. Добавить CI workflow и зафиксировать проверку `go test ./...` на Go 1.23.
4. Прогнать `go test ./...` локально и убедиться, что нет регрессий.

## Риски

- Риск: в коде/зависимостях используется функциональность, требующая Go > 1.23.
  Mitigation: выявить на раннем шаге через CI на 1.23; если не исправимо без переписывания — поднять DEC-001 (и синхронно обновить конституцию/доки).

## Rollout и compatibility

- Изменение влияет на требования окружения; для пользователей это “смягчение” (понижение минимума) или фиксация фактического минимума.
- Если минимум повышается (в случае выявления несовместимости) — это документируется явно в docs/README.

## Проверка

- Локально: `go test ./...`
- CI: `go test ./...` на Go 1.23
