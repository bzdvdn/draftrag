# Зафиксировать целевую версию Go (минимальную) и выровнять артефакты

## Scope Snapshot

- In scope: зафиксировать минимально поддерживаемую версию Go для библиотеки и привести `go.mod`, документацию и CI (если есть/будет) к одному значению, чтобы пользователи не упирались в неожиданный toolchain.
- Out of scope: рефакторинг кода ради новых фич Go, массовая замена зависимостей, а также изменение публичного API draftRAG, не связанное с совместимостью по Go.

## Цель

Сделать целевую версию Go понятной и непротиворечивой: пользователь видит одно и то же требование в `go.mod`, `.speckeep/constitution.md` и документации, а автоматические проверки подтверждают, что проект собирается и тестируется на минимально заявленной версии.

## Основной сценарий

1. Пользователь открывает README/docs и видит “минимальная версия Go: X”.
2. Пользователь делает `go get`/`go test` и не получает неожиданного требования скачивать toolchain “выше X” из-за `go.mod`.
3. В репозитории есть автоматическая проверка (CI или локальный target), которая гарантирует сборку/тесты на Go X.

## Scope

- Определить и зафиксировать **минимально поддерживаемую** версию Go для draftRAG (каноническое значение).
- Привести к нему:
  - `go.mod` (директива `go ...` и связанные с ней эффекты),
  - `.speckeep/constitution.md`,
  - README/docs (включая `docs/getting-started.md`).
- Добавить/обновить CI-проверку (если CI отсутствует — добавить минимальный workflow), которая прогоняет `go test ./...` на минимальной версии.

## Контекст

- В `go.mod` сейчас указано `go 1.23.x`, а в `docs/getting-started.md` — “Минимальная версия Go: 1.23”.
- Для библиотек несогласованность версий приводит к “неочевидным” ошибкам у пользователей (особенно из-за механики toolchain в новых версиях Go).

## Требования

- RQ-001 Проект ДОЛЖЕН иметь одно каноническое значение “минимальная версия Go” (например, `1.23`), используемое во всех публичных артефактах.
- RQ-002 `go.mod` ДОЛЖЕН быть выровнен под минимальную версию Go так, чтобы пользователю не требовался toolchain выше заявленного минимума для сборки/тестов библиотеки. При этом директива `toolchain` в `go.mod` **допускается** (для удобства разработки), но не должна менять/повышать заявленный минимум.
- RQ-003 Документация ДОЛЖНА содержать одинаковое требование к версии Go (без конфликтов между README, docs и constitution).
- RQ-004 ДОЛЖНА существовать автоматическая проверка (CI или эквивалентная в репозитории), подтверждающая `go test ./...` на минимальной версии Go.

## Вне scope

- Поддержка Go ниже минимально заявленной версии (например, 1.20 и ниже).
- Добавление новых фич языка Go ради “обновления версии”.
- Переписывание тестов/кода на новые пакеты стандартной библиотеки, если это не требуется для совместимости.

## Критерии приемки

### AC-001 `go.mod` соответствует целевой минимальной версии Go

- Почему это важно: `go.mod` — первичный источник правды для toolchain и совместимости.
- **Given** пользователь подключает модуль draftRAG
- **When** он собирает и запускает тесты на минимально заявленной версии Go
- **Then** `go test ./...` проходит без требования toolchain выше минимума
- Evidence: `go.mod` отражает целевую версию; тесты проходят на минимальной версии.

### AC-002 Документация согласована

- Почему это важно: docs — точка входа, откуда пользователи копируют требования.
- **Given** пользователь читает `.speckeep/constitution.md`, README и getting-started
- **When** он ищет минимальную версию Go
- **Then** везде указано одно и то же значение и не встречаются противоречивые числа
- Evidence: обновлены `.speckeep/constitution.md` (если нужно), `README.md`, `docs/getting-started.md`.

### AC-003 Есть автоматическая проверка минимальной версии

- Почему это важно: без проверки версия “поплывёт” со временем.
- **Given** новый PR меняет код или зависимости
- **When** запускаются проверки
- **Then** сборка/тесты подтверждаются на минимальной версии Go
- Evidence: CI/workflow (или репо-скрипт) гоняет `go test ./...` на минимальной версии.

## Допущения

- Кодовая база не использует возможности Go выше целевого минимума (или может быть приведена к совместимой форме без изменения публичного API).
- В репозитории допускается добавление минимального CI (например, GitHub Actions), если он отсутствует.

## Краевые случаи

- Разница “минимальная поддерживаемая версия” vs “рекомендованная версия для разработки”: если вводится рекомендуемая версия, она не должна ломать сборку на минимальной.
- Локальная сборка с более новым Go не должна заставлять downstream-пользователя скачивать новый toolchain без явного согласия.
