# Compatibility & Support Policy

## Scope Snapshot

- In scope: добавить единый, публичный документ политики совместимости и поддержки: поддерживаемые версии Go, статус backend’ов (vector stores / LLM / embedders), гарантия стабильности публичного API (semver), окна поддержки и правила депрекации.
- Out of scope: изменение реализации backend’ов, добавление новых провайдеров, изменение semver-схемы проекта, юридические SLA/поддержка по контракту.

## Цель

Пользователи (стартапы и энтерпрайз) должны быстро понимать “что поддерживается и насколько стабильно”: какие версии Go нужны, какие backend’ы production-ready, какие возможности доступны (streaming, filters, hybrid), как долго поддерживаются минорные версии и как будут объявляться breaking changes.

## Основной сценарий

1. Пользователь выбирает draftRAG для production и хочет оценить риски.
2. Пользователь открывает README и ищет “compatibility/support policy”.
3. Пользователь видит ссылку на документ, где есть таблицы совместимости и краткие правила поддержки/депрекации.
4. Пользователь использует документ для выбора backend’а и планирования апгрейда.

## Scope

- Создать документ (в `docs/`) “Compatibility & Support Policy” на русском языке.
- Добавить ссылку на документ в `README.md`.
- Документ должен включать:
  - поддерживаемые версии Go и политика их обновления;
  - матрицу совместимости backend’ов (vector store / embedder / LLM) и статусы (например, stable/experimental/deprecated);
  - матрицу поддержки возможностей (filters, hybrid, streaming, retries/CB, cache, observability hooks/OTel);
  - semver правила для публичного API `pkg/draftrag` и правила депрекации (как долго держим deprecated API, как объявляем).

## Контекст

- В конституции указан минимальный Go (1.23+) и акцент на стабильность публичного API.
- README уже перечисляет production-ready компоненты и примеры, но нет единого “контракта поддержки” и статусов.
- Проект имеет несколько backend’ов, которые отличаются по возможностям и эксплуатационным нюансам; пользователям важно видеть это явно.

## Требования

- RQ-001 ДОЛЖЕН существовать единый документ политики совместимости и поддержки в `docs/` (русский).
- RQ-002 Документ ДОЛЖЕН явно фиксировать минимальную версию Go и правило обновления (например, “поддерживаем N последних minor Go” или “минимум = X, пересматриваем раз в квартал”).
- RQ-003 Документ ДОЛЖЕН содержать таблицу совместимости backend’ов (vector stores / embedders / LLM) со статусами поддержки.
- RQ-004 Документ ДОЛЖЕН содержать таблицу возможностей (например, filters/hybrid/streaming/observability) с отметкой поддержки по backend’ам там, где это применимо.
- RQ-005 Документ ДОЛЖЕН фиксировать semver правила и процесс депрекации для `pkg/draftrag` (минимум: срок поддержки deprecated API и способ коммуникации).
- RQ-006 В `README.md` ДОЛЖНА быть ссылка на этот документ.

## Вне scope

- Переписывание или изменение поведения backend’ов ради соответствия матрице.
- Добавление нового CI-матриц тестирования по всем backend’ам (это отдельная фича).
- Юридические гарантийные обязательства поддержки.

## Критерии приемки

### AC-001 Документ политики добавлен и доступен из README

- Почему это важно: пользователи должны находить политику без поиска по репозиторию.
- **Given** пользователь читает `README.md`
- **When** он ищет информацию о поддержке/совместимости
- **Then** он находит ссылку на документ и может открыть его в `docs/`
- Evidence: файл документа существует в `docs/`, а `README.md` содержит явную ссылку на него

### AC-002 Политика Go/semver/депрекации описана без двусмысленностей

- Почему это важно: апгрейды и планы миграций зависят от стабильности API.
- **Given** пользователь планирует использовать `pkg/draftrag` в production
- **When** он читает документ
- **Then** он видит: минимальную версию Go, правило поддержки версий и явные правила semver/депрекации (сроки и коммуникация)
- Evidence: в документе есть отдельная секция с bullet-правилами и конкретными сроками/формулировками

### AC-003 Матрицы совместимости и возможностей присутствуют и пригодны для выбора backend’а

- Почему это важно: выбор backend’а часто определяется поддержкой фильтров/гибрида/стриминга.
- **Given** пользователь сравнивает backend’ы
- **When** он открывает матрицу в документе
- **Then** он видит таблицу backend’ов со статусами и отдельную таблицу возможностей, позволяющие принять решение без чтения кода
- Evidence: в документе есть минимум 2 таблицы: “Backends vs Status” и “Features vs Backends/Support”

## Допущения

- Статусы поддержки (stable/experimental/deprecated) — это документационный контракт, который может меняться по мере развития проекта, но должен обновляться в релизных notes.
- Поддержка “возможностей” в матрице — best-effort отражение текущего состояния публичного API и документации, без обещаний SLA.

## Краевые случаи

- Возможность доступна только для части backend’ов (например, hybrid — только pgvector): это должно быть явно отмечено в таблице.
- Deprecated API: документ должен описывать как минимум “как долго” и “как объявляем”, чтобы не ломать enterprise процессы.
