# PII Guardrails — PII detection + redaction на входе/выходе pipeline

## Scope Snapshot

- In scope: автоматическое обнаружение и цензурирование PII (персональных данных) в содержимом документов при индексации и в результатах при retrieval/answer.
- Out of scope: ML-based PII-детекция; сторонние PII-API как обязательное требование; streaming-режим; модификация метаданных документов.

## Цель

Разработчик, использующий draftRAG, получает возможность автоматически цензурировать PII в тексте документов на этапе индексации и в результатах поиска/ответа без внедрения собственных фильтров. Успех определяется наличием конфигурируемого PII-детектора (встроенные pattern-правила + поддержка кастомных), интеграцией с Index/Query/Retrieve/Answer и набором тестов для каждой категории PII.

## Основной сценарий

1. Пользователь создаёт Pipeline с опцией PII-детектора, указывая нужные категории (email, телефон, SSN, номер карты).
2. При вызове `Pipeline.Index` содержимое каждого документа проходит через детектор: обнаруженные PII заменяются на `<redacted>` перед сохранением в store.
3. При вызове `Pipeline.Query`, `Pipeline.Retrieve` или `Pipeline.Answer` результаты retrieval перед возвратом проходят через тот же детектор.
4. Если детектор не сконфигурирован — PII-обработка не применяется, поведение pipeline не меняется.

## User Stories

- P1 Story: Разработчик включает PII-guardrails через опцию pipeline и видит, что email/телефон/SSN в проиндексированных документах и в результатах Query заменены на `<redacted>`.
- P2 Story: Разработчик подключает кастомный детектор для своего типа PII (напр., passport numbers), и pipeline применяет его наравне со встроенными.

## MVP Slice

Встроенные pattern-детекторы для email, телефона (E.164), SSN + интеграция с Index и Query. AC-001, AC-002, AC-004.

## First Deployable Outcome

Пример в директории `examples/` с InMemoryStore, двумя документами с тестовыми PII, и вызовом Query — вывод подтверждает, что PII заменён на `<redacted>`. Работает без внешних зависимостей.

## Scope

- PII-детекция в содержимом документов на входе Index
- PII-детекция в содержимом retrieval-результатов на выходе Query, Retrieve, Answer
- Встроенные pattern-детекторы: email, телефон (E.164), SSN, номер кредитной карты
- Возможность подключения кастомного PII-детектора
- PII-детекция в переписанных запросах (RewrittenQuery) перед использованием для retrieval
- PipelineOption для включения/конфигурации PII-guardrails
- Backward compatibility: отсутствие детектора не меняет поведение

## Контекст

- В `internal/domain/redaction.go` уже есть базовые `RedactSecret`/`RedactSecrets` — механизм замены
- Pipeline поддерживает опции через `PipelineOptions` — новая опция встраивается в существующую механику
- Feature не требует внешних зависимостей (pattern-детекция через stdlib)

## Зависимости

- `none` — feature реализуется стандартной библиотекой Go (regexp) + существующие domain-типы

## Требования

- RQ-001 Pipeline ДОЛЖЕН цензурировать PII в `Document.Content` при вызове Index, если PII-детектор сконфигурирован.
- RQ-002 Pipeline ДОЛЖЕН цензурировать PII в `RetrievalResult.Content` при вызове Query, Retrieve и Answer, если PII-детектор сконфигурирован.
- RQ-003 Система ДОЛЖНА предоставлять встроенные PII-детекторы для email, телефона, SSN и номера кредитной карты.
- RQ-004 Система ДОЛЖНА позволять пользователю реализовать и подключить кастомный PII-детектор через Go-интерфейс.
- RQ-005 Поведение Index/Query/Retrieve/Answer НЕ ДОЛЖНО меняться при отсутствии сконфигурированного PII-детектора.
- RQ-006 Категории PII ДОЛЖНЫ быть независимо включаемыми/отключаемыми.
- RQ-007 Pipeline ДОЛЖЕН цензурировать PII в `RewrittenQuery` перед использованием для retrieval, если PII-детектор сконфигурирован.

## Вне scope

- PII-детекция в streaming-режиме (Stream, StreamSources, StreamCite)
- ML-based или API-based PII-детекция
- PII-детекция в метаданных документов (Document.Metadata)

- Автоматическое обнаружение PII при загрузке конфигурации без явного включения

## Критерии приемки

### AC-001 PII redaction при индексации

- Почему это важно: документы с PII не должны сохранять чувствительные данные в векторном store.
- **Given** Pipeline сконфигурирован с PII-детектором (email + телефон)
- **When** вызывается `Pipeline.Index` с документом, содержащим email `user@example.com` и телефон `+1-555-123-4567`
- **Then** оба вхождения заменены на `<redacted>` в документе перед сохранением
- Evidence: извлечение документа из store (через Retrieve по ID) показывает `<redacted>` вместо PII

### AC-002 PII redaction при retrieval

- Почему это важно: пользователь не должен получать PII в результатах поиска.
- **Given** Pipeline с PII-детектором, в store есть документ с PII-содержимым
- **When** вызывается `Pipeline.Query`
- **Then** все `RetrievalResult` в ответе содержат `<redacted>` вместо PII в поле Content
- Evidence: вывод Query не содержит оригинальных PII-значений

### AC-003 PII redaction при Answer

- Почему это важно: ответ LLM строится на контексте, который может содержать PII.
- **Given** Pipeline с PII-детектором, в store есть документ с PII
- **When** вызывается `Pipeline.Answer`
- **Then** все `RetrievalResult` в `AnswerResponse` содержат `<redacted>` вместо PII
- Evidence: AnswerResponse.RetrievalResults не содержат оригинальных PII-значений

### AC-004 Выборочное включение категорий PII

- Почему это важно: разработчик может контролировать, какие типы PII обрабатывать.
- **Given** PipelineOptions сконфигурирован только с email-детектором
- **When** индексируется документ с email и телефоном
- **Then** только email заменён на `<redacted>`, телефон остался нетронутым
- Evidence: проверка сохранённого документа в store

### AC-005 Кастомный PII-детектор

- Почему это важно: разработчик может расширять детекцию под специфичные данные.
- **Given** пользователь реализовал кастомный детектор для passport numbers
- **When** Pipeline создаётся с этим кастомным детектором и вызывается Index
- **Then** passport numbers в документах заменены на `<redacted>`
- Evidence: извлечение документа из store

### AC-006 Backward compatibility

- Почему это важно: существующие pipelines не должны ломаться.
- **Given** Pipeline создан без PII-детектора (`nil`)
- **When** вызывается Index или Query
- **Then** содержимое документов/результатов не изменено
- Evidence: содержимое до и после вызова идентично

### AC-007 PII redaction в переписанных запросах

- Почему это важно: PII из пользовательского запроса не должен попадать в retrieval через переписанный запрос.
- **Given** Pipeline с PII-детектором и QueryRewriter, в store есть документ
- **When** вызывается `Pipeline.Query` с запросом, содержащим PII
- **Then** все `RewrittenQuery`, сгенерированные для этого запроса, не содержат PII
- Evidence: проверка переписанных запросов через инструментирование или перехват

## Допущения

- PII-детектор применяется только к полю `Document.Content` и `RetrievalResult.Content`; метаданные не обрабатываются
- Для замены используется маркер `<redacted>` (из существующего `redactedMarker`)
- PipelineOption для PII-детектора имеет тип интерфейса, определённого в domain
- Все встроенные детекторы реализуются через `regexp` из стандартной библиотеки Go

## Критерии успеха

- SC-001 Добавление PII-детектора не увеличивает latency Index более чем на 5% для документов до 10 KB (измерение через benchmark)
- SC-002 Покрытие встроенных pattern-детекторов: минимум 3 тест-кейса на каждый pattern (валидный, невалидный, граничный)

## Краевые случаи

- Документ без PII — детектор не модифицирует содержимое
- PII как часть более длинного токена (напр., `contact@example.com.net`) — pattern должен корректно обрабатывать границы
- Пустой Content — no-op
- Перекрывающиеся PII-совпадения — порядок применения детекторов детерминирован
- Очень длинный документ (>1 MB) — детектор не должен падать по памяти

## Открытые вопросы

- Решено: IP-адреса и даты рождения — не входят в MVP.
- Решено: PII-детектор применяется к RewrittenQuery (RQ-007, AC-007).
- Решено: метрики/log не обязательны.
