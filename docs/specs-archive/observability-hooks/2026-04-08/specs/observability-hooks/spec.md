# Observability: хуки/метрики для pipeline стадий (v1)

## Scope Snapshot

- In scope: добавить лёгкий механизм наблюдаемости (hooks) для стадий pipeline: chunking, embed, search, generate; с длительностями и ошибками.
- Out of scope: встроенный HTTP endpoint `/metrics`, конкретные метрик-бэкенды (Prometheus/OpenTelemetry), логгер как зависимость в domain.

## Контекст

- Сейчас pipeline “чёрный ящик”: пользователю трудно понять, где именно тратится время (embed/search/generate) и на какой стадии происходят ошибки.
- Для production-использования нужна минимальная наблюдаемость без навязывания конкретного стека (Prometheus/OTel/loggers), чтобы:
  - измерять latency по стадиям,
  - считать ошибки по стадиям,
  - строить трейсинг/метрики на своей стороне.

## Цель

Дать пользователю возможность измерять latency и ошибки по стадиям RAG без форка библиотеки и без жёсткой привязки к конкретному observability стеку.

## Основной сценарий

1. Пользователь передаёт hooks/observer в Pipeline config.
2. При выполнении Index/Query/Answer вызываются callbacks “start/end” (или end с duration).
3. Пользователь экспортирует метрики/трейсы своим способом.

## Требования

- RQ-001 ДОЛЖЕН существовать интерфейс hooks (в domain или pkg) без внешних зависимостей.
- RQ-002 Hooks ДОЛЖНЫ покрывать стадии:
  - chunking (если включён),
  - embed,
  - vector search,
  - LLM generate.
- RQ-003 Hooks ДОЛЖНЫ получать `context.Context` и информацию об ошибке (если есть).
- RQ-004 Hooks НЕ ДОЛЖНЫ ломать поведение по умолчанию (nil hooks = no-op).
- RQ-005 Тесты ДОЛЖНЫ проверять, что hooks вызываются ожидаемое число раз и в правильном порядке (без внешней сети).

## Критерии приемки

### AC-001 Hooks вызываются на стадиях Answer

- **Given** pipeline с подключенными hooks
- **When** выполняется Answer
- **Then** hooks получают события embed/search/generate (и chunking при наличии)

### AC-002 Nil hooks не влияет

- **Given** hooks не задан
- **When** выполняются операции pipeline
- **Then** поведение идентично текущему, без паник

## Открытые вопросы

- Делать ли hooks “sync” или разрешить async? (v1: sync, ответственность за лёгкость на пользователе)
