# Eval harness: только retrieval метрики (без качества генерации)

## Scope Snapshot

- In scope: расширение eval harness дополнительными retrieval-метриками и улучшением инфраструктуры для оценки качества поиска без участия генерации ответов
- Out of scope: метрики качества генерации (answer quality, faithfulness, relevance), LLM-based evaluation

## Цель

Разработчики и пользователи draftRAG получают инструмент для оценки качества retrieval-компонента RAG-системы без зависимости от LLM-генерации. Это позволяет быстро и дёшево оценивать эффективность векторного поиска, chunker-а и embedding-модели по классическим IR-метрикам. Успех виден по наличию стабильного набора retrieval-метрик (Hit@K, MRR, NDCG, Precision/Recall) и удобной инфраструктуре для запуска eval-прогонов.

## Основной сценарий

1. Пользователь подготавливает датасет с вопросами и ожидаемыми документами (parent IDs).
2. Пользователь конфигурирует harness (выбор метрик, параметры topK).
3. Harness прогоняет retrieval через Pipeline для каждого кейса датасета.
4. Harness вычисляет агрегированные retrieval-метрики по всем кейсам.
5. Пользователь получает отчёт с метриками и детальными результатами по каждому кейсу.

## Scope

- Расширение набора retrieval-метрик (добавление NDCG, Precision@K, Recall@K)
- Улучшение структуры отчёта для детального анализа
- Поддержка конфигурации выбора метрик через Options
- Инфраструктура для сериализации/десериализации датасетов и результатов
- Валидация входных данных и детализация ошибок

## Контекст

- Существующий `pkg/draftrag/eval/` содержит базовый harness с Hit@K и MRR
- RetrievalRunner интерфейс уже абстрагирует retrieval от генерации
- Оценка генерации требует дорогих LLM-вызовов и не всегда нужна при отладке retrieval
- DraftRAG следует принципу минимальной конфигурации и интерфейсной абстракции

## Требования

- RQ-001 Harness ДОЛЖЕН поддерживать вычисление NDCG@K для оценки ранжирования с учётом релевантности
- RQ-002 Harness ДОЛЖЕН поддерживать вычисление Precision@K и Recall@K для оценки точности и полноты поиска
- RQ-003 Harness ДОЛЖЕН позволять конфигурировать набор вычисляемых метрик через Options
- RQ-004 Harness ДОЛЖЕН сохранять детальные результаты по каждому кейсу для дебага и анализа
- RQ-005 Harness ДОЛЖЕН валидировать входные данные (непустые вопросы, корректные ID) и возвращать понятные ошибки
- RQ-006 Harness ДОЛЖЕН поддерживать сериализацию Report в JSON для интеграции с CI/CD

## Вне scope

- Метрики качества генерации (faithfulness, answer relevance, factual correctness)
- LLM-based evaluation (асессор через LLM)
- End-to-end evaluation retrieval + generation
- Метрики на основе эмбеддингов (embedding similarity)
- Визуализация результатов (dashboards, графики)

## Критерии приемки

### AC-001 Поддержка NDCG@K

- Почему это важно: NDCG учитывает позицию релевантных документов и градации релевантности, что важнее для ранжирования чем бинарный Hit@K
- **Given** датасет с вопросами и ExpectedParentIDs, где каждый ID имеет вес релевантности (опционально)
- **When** пользователь запускает harness с включённой метрикой NDCG@K
- **Then** отчёт содержит NDCG@K для каждого кейса и агрегированное значение по датасету
- Evidence: в Report.Metrics присутствует поле NDCG с корректным значением от 0 до 1

### AC-002 Поддержка Precision@K и Recall@K

- Почему это важно: Precision и Recall - классические IR-метрики для оценки точности и полноты поиска
- **Given** датасет с вопросами и множеством ExpectedParentIDs
- **When** пользователь запускает harness с включёнными метриками Precision@K и Recall@K
- **Then** отчёт содержит Precision@K и Recall@K для каждого K и агрегированные значения
- Evidence: в Report.Metrics присутствуют поля Precision и Recall с корректными значениями от 0 до 1

### AC-003 Конфигурируемый набор метрик

- Почему это важно: разные сценарии требуют разных метрик; гибкость позволяет избегать лишних вычислений
- **Given** пользователь создаёт Options с выбранным набором метрик (например, только Hit@K и MRR)
- **When** пользователь запускает harness с этими Options
- **Then** harness вычисляет только запрошенные метрики, остальные поля отчёта остаются нулевыми или отсутствуют
- Evidence: при отключённой метрике соответствующее поле в Report.Metrics имеет значение 0 или не вычисляется

### AC-004 Детализация результатов по кейсам

- Почему это важно: необходимо понимать, на каких конкретных кейсах retrieval работает плохо
- **Given** датасет из нескольких кейсов
- **When** пользователь запускает harness
- **Then** Report.Cases содержит детальные результаты для каждого кейса (Found, Rank, RetrievedParentIDs, значения метрик)
- Evidence: Report.Cases[i] содержит поля с результатами для i-го кейса

### AC-005 Валидация входных данных

- Почему это важно: некорректные входные данные должны детектироваться рано с понятными ошибками
- **Given** датасет с кейсом, у которого Question пустой или ExpectedParentIDs содержит пустые строки
- **When** пользователь запускает harness
- **Then** harness возвращает ошибку с описанием проблемы до начала retrieval
- Evidence: функция Run возвращает error с текстом, указывающим на некорректный кейс

### AC-006 Сериализация Report в JSON

- Почему это важно: необходимо интегрировать eval в CI/CD и сохранять результаты для исторического анализа
- **Given** успешно выполненный harness с Report
- **When** пользователь вызывает метод сериализации Report
- **Then** Report преобразуется в валидный JSON со всеми метриками и деталями кейсов
- Evidence: Report реализует метод MarshalJSON или функцию для сериализации, результат валиден по json.Unmarshal

## Допущения

- ExpectedParentIDs уникальны в рамках одного кейса (дубликаты игнорируются)
- Релевантность по умолчанию бинарная (1 для всех ExpectedParentIDs), если не указаны веса
- Вес релевантности для NDCG опционален; по умолчанию все релевантные документы имеют вес 1
- K для метрик @K определяется из TopK кейса или DefaultTopK из Options
- Dataset помещается в память (нет стриминга для больших датасетов)

## Критерии успеха

- SC-001 Вычисление всех метрик для датасета из 1000 кейсов завершается менее чем за 10 секунд
- SC-002 Потребление памяти при прогоне датасета из 10000 кейсов не превышает 500MB
- SC-003 Unit-тесты покрывают все функции вычисления метрик (NDCG, Precision, Recall)

## Краевые случаи

- Пустой датасет (0 кейсов): harness возвращает пустой Report с нулевыми метриками без ошибки
- ExpectedParentIDs пустой: кейс считается не найденным (Found=false, Rank=0)
- TopK=0 в кейсе и Options: используется дефолтное значение 5
- RetrievedParentIDs содержат дубликаты: дубликаты игнорируются при вычислении метрик
- ExpectedParentIDs содержат ID, которых нет в индексе: это нормально, метрики вычисляются по retrieved
