# Кэширование векторов (Embedding Cache)

## Scope Snapshot

- In scope: Интерфейс-обёртка `EmbedderCache` над `Embedder` с LRU in-memory кэшем и опциональным Redis-бэкендом.
- Out of scope: Persistent storage для in-memory кэша, distributed locking, cache warming strategies.

## Цель

Разработчики, использующие draftRAG, получают снижение затрат на API эмбеддингов при повторной индексации документов и повторяющихся запросах за счёт кэширования векторов по хэшу входного текста.

## Основной сценарий

1. **Стартовая точка**: Пользователь настраивает `EmbedderCache` как обёртку над существующим `Embedder` (например, OpenAI).
2. **Первый вызов**: При вызове `Embed(ctx, "текст документа")` кэш вычисляет хэш текста, проверяет наличие в кэше — не находит, делегирует вызов базовому `Embedder`, сохраняет результат в кэш и возвращает вектор.
3. **Повторный вызов**: При повторном вызове с тем же текстом кэш находит вектор по хэшу и возвращает без обращения к API.
4. **Ограничение кэша**: При превышении лимита LRU in-memory кэша evict'ятся наименее используемые записи; при включённом Redis кэш читает/пишет в Redis как второй уровень.

## Scope

- Интерфейс `EmbedderCache`, реализующий `domain.Embedder`
- In-memory LRU реализация кэша с настраиваемым размером
- Опциональный Redis-бэкенд как second-level cache
- Хэширование входного текста (SHA-256) для ключа кэша
- Поддержка TTL для записей кэша
- Thread-safe операции кэша
- Интеграция с существующим `domain.Embedder` интерфейсом
- Метрики/статистика попаданий в кэш (hits/misses)

## Контекст

- Интерфейс `domain.Embedder` имеет метод `Embed(ctx context.Context, text string) ([]float64, error)`
- Существующие реализации `Embedder` находятся в `internal/infrastructure/embedder/`
- Конституция требует thread-safety и поддержки `context.Context`
- Векторы имеют фиксированную размерность (зависит от модели), хранятся как `[]float64`
- Хэш текста используется как ключ кэша для консистентности между запусками

## Требования

- **RQ-001** `EmbedderCache` ДОЛЖЕН реализовывать интерфейс `domain.Embedder`
- **RQ-002** `EmbedderCache` ДОЛЖЕН принимать базовый `Embedder` в конструкторе
- **RQ-003** `EmbedderCache` ДОЛЖЕН использовать SHA-256 хэш текста как ключ кэша
- **RQ-004** In-memory реализация ДОЛЖНА использовать LRU-алгоритм с настраиваемым размером
- **RQ-005** Все операции кэша ДОЛЖНЫ быть thread-safe
- **RQ-006** Redis-бэкенд ДОЛЖЕН сериализовать векторы в компактный формат (msgpack/binary)
- **RQ-007** Redis-бэкенд ДОЛЖЕН поддерживать TTL для записей
- **RQ-008** Кэш ДОЛЖЕН предоставлять методы для получения статистики (hits, misses, hit rate)
- **RQ-009** При ошибке Redis кэш ДОЛЖЕН fallback'иться к in-memory и логировать warning

## Вне scope

- Persistent storage для in-memory кэша (сохранение между рестартами)
- Distributed locking для Redis (конкурентные записи)
- Cache warming / prefetch стратегии
- Инвалидация кэша по паттерну или тегам
- Мульти-уровневые кэши более 2 уровней
- Compression для in-memory хранения
- Кэширование на уровне HTTP/transport

## Критерии приемки

### AC-001 Базовое кэширование in-memory

- **Почему важно**: Основной сценарий экономии API-вызовов при повторной индексации
- **Given** `EmbedderCache` создан с базовым `Embedder` и размером кэша 100
- **When** вызывается `Embed(ctx, "тестовый текст")` дважды
- **Then** первый вызов обращается к базовому `Embedder`, второй возвращает закэшированный результат без API-вызова
- **Evidence**: Счётчик вызовов базового `Embedder` равен 1; статистика показывает 1 hit, 1 miss

### AC-002 LRU eviction при переполнении

- **Почему важно**: Предотвращает неограниченный рост памяти
- **Given** `EmbedderCache` с размером кэша 2 и тремя различными текстами
- **When** вызывается `Embed` для всех трёх текстов (первый текст вызывался раньше всех)
- **Then** первый текст evict'ится из кэша; повторный вызов первого текста требует API-вызова
- **Evidence**: Размер кэша не превышает 2; статистика показывает корректное количество evictions

### AC-003 Thread-safety

- **Почему важно**: `Embedder` используется из многих горутин в production
- **Given` `EmbedderCache` с размером 100, 10 горутин, каждая вызывает `Embed` 100 раз с разными текстами
- **When** все горутины завершают работу
- **Then** нет race conditions, паник или data corruption; статистика корректна
- **Evidence**: Тест проходит с `-race` флагом; итоговая статистика совпадает с ожидаемой

### AC-004 Redis fallback при недоступности

- **Почему важно**: Система должна работать даже при сбоях Redis
- **Given** `EmbedderCache` с включённым Redis и недоступным Redis-сервером
- **When** вызывается `Embed(ctx, "текст")`
- **Then** кэш fallback'ится к in-memory, выполняет API-вызов, возвращает результат без ошибки
- **Evidence**: Лог содержит warning о недоступности Redis; результат корректен; нет panic

### AC-005 Redis как second-level cache

- **Почему важно**: Разделяемый кэш между инстансами приложения
- **Given** `EmbedderCache` с Redis, ключ "abc" существует в Redis
- **When** вызывается `Embed` с текстом, дающим хэш "abc"
- **Then** вектор читается из Redis без обращения к базовому `Embedder`
- **Evidence**: Счётчик вызовов базового `Embedder` равен 0; статистика показывает 1 hit

### AC-006 Хэш консистентности

- **Почему важно**: Одинаковые тексты должны давать одинаковые ключи кэша
- **Given` два идентичных текста в разных строках с разным allocation
- **When** вычисляется хэш для обоих
- **Then** хэши идентичны
- **Evidence**: Хэши равны; `bytes.Equal` показывает идентичность

### AC-007 Статистика кэша

- **Почему важно**: Наблюдаемость эффективности кэширования
- **Given` `EmbedderCache` после серии вызовов (5 hits, 3 misses)
- **When` запрашивается статистика через метод `Stats()`
- **Then** возвращается структура с Hits=5, Misses=3, HitRate=0.625
- **Evidence**: Значения статистики соответствуют фактическим вызовам

## Допущения

- Входные тексты для эмбеддинга имеют разумный размер (<1MB); хэширование не является bottleneck
- SHA-256 коллизии пренебрежимо малы для практического использования
- Векторы имеют размерность от 256 до 4096 float64 (2-32KB на запись)
- Redis используется как опциональный компонент; наличие Redis не обязательно для работы
- Пользователь сам управляет размером in-memory кэша исходя из доступной памяти
- Базовый `Embedder` возвращает детерминированные результаты для одного текста

## Критерии успеха

- **SC-001** In-memory кэш с размером 10k записей занимает <500MB RAM (типичный вектор 1536 float64 ≈ 12KB)
- **SC-002** Hit latency <1μс для in-memory кэша (95-й перцентиль)
- **SC-003** Miss latency не более чем на 5% выше latency базового `Embedder`
- **SC-004** Redis hit latency <5мс (95-й перцентиль) при RTT <2мс до Redis

## Краевые случаи

- Пустой текст: хэш от пустой строки должен работать корректно
- Очень большой текст (>1MB): вычисление хэша может быть expensive; возможно ограничение размера ключа
- Нулевой размер кэша: LRU кэш с size=0 всегда miss, работает как прокси
- Redis timeout: должен приводить к fallback, не к ошибке пользователю
- Контекст отменён: операция должна уважать `ctx.Done()`
