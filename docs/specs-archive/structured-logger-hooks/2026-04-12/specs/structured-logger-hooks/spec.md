# Structured logger hooks (замена log.Printf)

## Scope Snapshot

- In scope: добавить опциональный структурированный логгер/хуки в публичные опции, чтобы библиотека не писала напрямую через `log.Printf` (в первую очередь кэш и retry-обёртки).
- Out of scope: полноценная система метрик/трейсинга (Prometheus/OpenTelemetry), гарантии доставки логов, единая конфигурация логирования на весь pipeline без явной передачи опций.

## Цель

Пользователь draftRAG может подключить свой структурированный логгер (или простой hook), чтобы события деградации/ошибок (например, Redis L2 недоступен; retry попытка; circuit breaker отклоняет вызов) фиксировались в наблюдаемом виде и в одном месте с логами приложения. При этом библиотека остаётся “тихой” по умолчанию и не пишет в глобальный `log` без явного согласия пользователя.

## Основной сценарий

1. Пользователь создаёт `CachedEmbedder` и/или `RetryEmbedder`/`RetryLLMProvider`, передавая опциональный логгер в их опции.
2. При ошибках инфраструктуры (Redis read/write, decode) или при retry/CB-событиях код вызывает логгер с контекстом и структурированными полями.
3. Если логгер не задан — поведение идентично текущему: нет прямых логов из библиотеки, ошибки обрабатываются как раньше.

## Scope

- Ввести публичный минимальный интерфейс структурированного логгера (или hook) в `pkg/draftrag`, пригодный для адаптации под `log/slog`, zap, logrus и т.п.
- Прокинуть логгер в кэш эмбеддингов (`CachedEmbedder`/`EmbedderCache`) и заменить прямые вызовы `log.Printf` на вызов логгера.
- Прокинуть логгер в retry-обёртки (`RetryEmbedder`, `RetryLLMProvider`) и логировать ключевые события (retry attempt, non-retryable error, circuit breaker rejection/transition) в структурированном виде.
- Добавить unit-тесты, подтверждающие вызовы логгера в ключевых ветках, без внешних сервисов.
- Обновить документацию (минимально: показать как подключить логгер).

## Контекст

- draftRAG — библиотека; прямые `log.Printf` изнутри библиотеки ухудшают наблюдаемость (невозможно унифицировать формат/уровни/политику) и могут “шуметь” в production.
- В коде уже есть события, которые полезно наблюдать (например, treat-as-miss при деградации Redis L2; retry попытки и circuit breaker), но пользователь сейчас не может централизованно собирать эти сигналы структурировано.
- В соответствии с конституцией, внешний инструмент (логгер приложения) должен подключаться через интерфейс и быть опциональным.

## Требования

- RQ-001 Публичный API ДОЛЖЕН предоставить опциональный структурированный логгер (или hook) без внешних зависимостей.
- RQ-002 По умолчанию (логгер не задан) библиотека НЕ ДОЛЖНА писать в глобальный `log` и НЕ ДОЛЖНА менять поведение существующих вызовов.
- RQ-003 Логгер ДОЛЖЕН принимать `context.Context` и структурированные поля (ключ/значение), чтобы интеграция с приложением была прямой.
- RQ-004 `CachedEmbedder`/`EmbedderCache` ДОЛЖЕН использовать логгер для событий деградации Redis L2 (read/write/decode) вместо `log.Printf`.
- RQ-005 `RetryEmbedder` и `RetryLLMProvider` ДОЛЖНЫ (при заданном логгере) логировать ключевые события retry/circuit breaker, не меняя семантику ошибок и ретраев.
- RQ-006 Логи ошибок в библиотеке ДОЛЖНЫ следовать принципу treat-as-miss / best-effort: логирование никогда не должно ломать основной путь выполнения.
- RQ-007 Новая функциональность ДОЛЖНА быть покрыта unit-тестами: проверка, что логгер вызывается в нужных ветках, и что при nil-логгере поведение не меняется.

## Вне scope

- Унификация логгера на уровне `Pipeline` без явных опций (т.е. “магическая” глобальная конфигурация).
- Замена `domain.Hooks` или расширение существующего интерфейса hooks (это сломает совместимость).
- Тонкая настройка уровней/семплирования/дедупликации логов.
- Экспорт метрик/трейсов.

## Критерии приемки

### AC-001 Обратная совместимость и no-op по умолчанию

- Почему это важно: библиотека не должна навязывать логирование и не должна ломать существующие интеграции.
- **Given** пользовательский код использует `CachedEmbedder` и `RetryEmbedder`/`RetryLLMProvider` без нового логгера
- **When** проект компилируется и запускает существующие unit-тесты
- **Then** поведение и сигнатуры существующих конструкторов остаются работоспособными, а библиотека не пишет напрямую через `log.Printf`
- Evidence: `go test ./...` проходит; в изменённых местах отсутствуют прямые `log.Printf`.

### AC-002 Кэш логирует деградацию Redis L2 структурированно

- Почему это важно: Redis недоступен/данные повреждены — это production сигнал, который должен попадать в логи приложения.
- **Given** включён Redis L2 и передан логгер
- **When** Redis `Get/Set` или decode возвращают ошибку
- **Then** вызывается логгер с уровнем `warn`/`error` и полями минимум `{component, operation, key_prefix?, err}`
- Evidence: unit-тест с fake logger подтверждает вызов и наличие ключевых полей.

### AC-003 Retry-обёртки логируют retry/CB события

- Почему это важно: без логов сложно понять, почему растёт латентность (ретраи) или почему запросы отклоняются (CB).
- **Given** `RetryEmbedder` или `RetryLLMProvider` сконфигурированы с логгером
- **When** происходит retry attempt или circuit breaker отклоняет выполнение
- **Then** логгер получает структурированное событие с полями `{component, operation, attempt?, rejected?, err?}`
- Evidence: unit-тест с контролируемым embedder/LLM, который форсирует retry/CB rejection, и fake logger, который фиксирует события.

### AC-004 Логирование не ломает основной поток

- Почему это важно: наблюдаемость не должна ухудшать надёжность.
- **Given** логгер задан, но его реализация возвращает panic/ошибку (или медленная)
- **When** библиотека пытается залогировать событие
- **Then** основной метод (`Embed`/`Generate`) не падает из-за логирования (логгер защищён/изолирован)
- Evidence: unit-тест с logger, который паникует, но операция завершается как предусмотрено (или корректно возвращает исходную ошибку).

### AC-005 Документация показывает подключение логгера

- Почему это важно: без примера пользователи не будут подключать наблюдаемость корректно.
- **Given** пользователь читает `README.md` или `docs/embedders.md`
- **When** он хочет подключить структурированное логирование
- **Then** он видит короткий пример, как передать логгер в опции (и/или адаптировать `log/slog`)
- Evidence: обновлённый раздел документации с минимальным примером.

## Допущения

- Пользовательская система логирования умеет принимать `context.Context` или может игнорировать его безопасно.
- Структурированные поля представлены парами ключ/значение; конкретный формат (JSON и т.п.) — ответственность приложения.

## Критерии успеха

- SC-001 Пользователь может централизованно собрать сигналы деградации (Redis/Retry/CB) в логах приложения без patch’ей библиотечного кода.

## Краевые случаи

- Логгер не задан (nil) → полный no-op.
- Логгер паникует → библиотека должна защититься и продолжить выполнение.
- Высокая частота событий retry → логирование должно оставаться best-effort (без гарантий).
