# Security redaction: секреты и PII в ошибках и логах

## Scope Snapshot

- In scope: обеспечить безопасное по умолчанию поведение: секреты (API keys / bearer tokens) и потенциально чувствительные фрагменты не должны утекать в `error.Error()` и в структурированные логи, которые генерирует библиотека.
- Out of scope: полноценный DLP/PII-сканер, автоматическая классификация текста, шифрование данных, секрет-менеджмент и политика хранения логов в приложении пользователя.

## Цель

Стартапы и энтерпрайз команды должны иметь уверенность, что при ошибках внешних провайдеров и при включённом логировании (cache/retry/circuit breaker) библиотека draftRAG не будет случайно выводить в ошибки/логи секреты (API ключи, токены авторизации) и не будет провоцировать утечки чувствительных данных через “слишком подробные” сообщения об ошибках.

## Основной сценарий

1. Пользователь настраивает провайдеры (LLM/Embedder/VectorStore) через options, в том числе передаёт `APIKey`/токены.
2. Внешний сервис отвечает ошибкой (non-2xx, сетевой сбой, rate limit) и возвращает body/сообщение, которое может содержать ключ/токен или кусок запроса.
3. Библиотека возвращает ошибку наверх и/или пишет structured log (например, Redis L2 или retry/CB).
4. Ошибка и лог не содержат секретов; сообщение остаётся полезным для диагностики (статус, код, стадия, retriable/non-retriable), но без утечек.

## Scope

- Редакция секретов в текстах ошибок, которые формирует библиотека для внешних провайдеров (LLM/Embedder/VectorStore), если там присутствуют известные секреты из options (например, `APIKey`) или заголовки авторизации.
- Редакция секретов в полях structured logging, которые библиотека сама пишет через `Logger` (cache/retry/circuit breaker).
- Документация: кратко описать, что библиотека редактирует и что остаётся ответственностью пользователя (например, не логировать сырые документы/запросы в своём приложении).

## Контекст

- В проекте уже есть структурированный `Logger` (и `SafeLog`) и места, где библиотека пишет логи (cache/retry/circuit breaker).
- В проекте уже есть тесты, которые проверяют редактирование APIKey в сообщениях ошибок для OpenAI-compatible LLM; это поведение нужно расширить/унифицировать на остальные провайдеры/поверхности, где актуально.
- Документация проекта на русском; публичные контракты должны оставаться ясными и проверяемыми.

## Требования

- RQ-001 Ошибки, возвращаемые публичным API, НЕ ДОЛЖНЫ содержать значения секретов, переданных пользователем через options (например, `APIKey`), если библиотека имела доступ к этим значениям.
- RQ-002 Логи, эмитируемые библиотекой через `Logger`, НЕ ДОЛЖНЫ содержать значения секретов, известных библиотеке (например, API keys/Authorization bearer tokens), ни в `msg`, ни в `fields`.
- RQ-003 Редакция ДОЛЖНА сохранять диагностическую ценность: статус/код/тип ошибки/operation/stage остаются в сообщении или полях.
- RQ-004 Редакция ДОЛЖНА быть best-effort и безопасной: отсутствие секрета, пустые значения и частичное совпадение не должны приводить к паникам или повреждению normal flow.
- RQ-005 README/доки ДОЛЖНЫ кратко описывать границы ответственности: библиотека редактирует известные ей секреты, но пользователь отвечает за собственные логи и содержимое документов/запросов.

## Вне scope

- Автоматическое обнаружение PII в произвольном тексте (именах, адресах, телефонах и т.п.).
- Скрытие/редакция “всего текста запроса/документа” (это может разрушить отладку и требует отдельной политики).
- Интеграция с секрет-менеджерами, ротация ключей, хранение секретов.
- Пересмотр архитектуры ошибок (введение новой иерархии ошибок) кроме точечных изменений для редактирования.

## Критерии приемки

### AC-001 Секреты не попадают в `err.Error()` для встроенных провайдеров

- Почему это важно: ошибки часто попадают в APM/лог-агрегаторы; утечки секретов — критический инцидент.
- **Given** пользователь создал провайдер (LLM/Embedder/VectorStore), передав `APIKey`/токен в options
- **When** провайдер возвращает ошибку, где исходное сообщение/response body содержит этот секрет
- **Then** ошибка, возвращаемая из публичного API draftRAG, не содержит значение секрета (заменено на `<redacted>` или эквивалент)
- Evidence: unit-тесты/интеграционные тесты на уровне `pkg/draftrag` покрывают минимум один LLM и один Embedder сценарий редактирования

### AC-002 Structured logging не содержит секретов

- Почему это важно: `Logger` предназначен для production; поля часто индексируются и распространяются.
- **Given** пользователь подключил `Logger` в `CacheOptions` или `RetryOptions`
- **When** библиотека логирует событие деградации/ошибки (например, Redis read/write failed, retry attempt failed, circuit breaker rejected)
- **Then** в `msg` и `fields` нет значений известных секретов (APIKey/Authorization bearer token)
- Evidence: unit-тест логгера-коллектора подтверждает, что в лог-событиях отсутствуют секреты при искусственно созданной ошибке

### AC-003 Документация описывает безопасные границы

- Почему это важно: без явных границ пользователи либо переоценят защиту, либо будут избегать observability.
- **Given** пользователь читает README/доки
- **When** он ищет, что редактируется и что нет
- **Then** он видит короткое описание: библиотека редактирует известные ей секреты; пользователь не должен логировать сырые документы/запросы без собственной политики
- Evidence: в README/документации есть секция/абзац про redaction и ответственность

## Допущения

- “Секрет” в рамках этой фичи = значения, которые библиотека получает напрямую через options (APIKey/токены) и которые могут попасть в error/log.
- Если секрет не был известен библиотеке (например, пользователь сам вложил его в текст документа), автоматическая редакция не гарантируется (вне scope).

## Краевые случаи

- Секрет может быть пустым или состоять из пробелов — редакция должна быть no-op.
- Секрет может встретиться в ошибке несколько раз — должны редактироваться все вхождения.
- Ошибка может быть wrapped (`fmt.Errorf("...: %w", err)`) — редакция должна применяться к итоговому сообщению, которое возвращается наружу/в лог.
