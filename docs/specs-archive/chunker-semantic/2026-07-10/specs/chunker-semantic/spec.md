# Semantic Chunker — семантическое разбиение документов на чанки

## Scope Snapshot

- In scope: новая реализация Chunker, которая разбивает документ на семантически связные чанки на основе косинусного сходства эмбеддингов предложений.
- Out of scope: замена BasicChunker, мультиязычный NLP-парсер предложений, LLM-based чанкинг.

## Цель

Разработчик RAG-пайплайна получает возможность разбивать документы на чанки, границы которых определяются семантической связностью, а не фиксированным числом рун. Успех фичи измеряется тем, что для типового документа чанки содержат тематически целостные фрагменты (одна тема на чанк), а метрики retrieval@k не падают относительно BasicChunker на равном числе чанков.

## Основной сценарий

1. Пользователь создаёт SemanticChunker через `NewSemanticChunker`, передавая Embedder, порог сходства и ограничения размера чанка.
2. При вызове `Chunk(ctx, doc)` чанкер разбивает `doc.Content` на предложения по правилам sentence boundary detection.
3. Чанкер группирует предложения в кандидаты, начиная с первого предложения.
4. Для каждого кандидата вычисляется эмбеддинг через переданный Embedder.
5. Косинусное сходство между эмбеддингом текущего кандидата и предыдущего чанка сравнивается с порогом: выше порога — предложение присоединяется к текущему чанку, ниже — начинается новый чанк.
6. Минимальный и максимальный размер чанка (в символах) соблюдаются: накопление продолжается до min размера, граница форсируется после max.
7. Результат — слайс domain.Chunk, где каждый чанк заканчивается на границе предложения.
8. При отмене контекста возвращается соответствующая ошибка.

## User Stories

- P1 (MVP): Как разработчик, я хочу получить семантический чанкер, который использует Embedder пайплайна и порог сходства, чтобы получать тематически целостные чанки без ручной настройки размера.
- P2 (расширение): Как разработчик, я хочу управлять минимальным/максимальным размером чанка, чтобы чанки не были слишком мелкими или слишком крупными для модели эмбеддинга.

## MVP Slice

P1: базовая реализация semantic chunker с sentence splitting по `.`/`!`/`?`, эмбеддингом через переданный Embedder, одним порогом сходства, min/max размером. Первичные AC: AC-001, AC-002, AC-003, AC-004.

## First Deployable Outcome

Юнит-тесты, демонстрирующие, что для документа с двумя тематическими блоками чанкер возвращает >= 2 чанков с корректными границами по предложениям, и что изменение порога влияет на количество чанков. Интеграция через `PipelineOptions.Chunker`.

## Scope

- Новая публичная функция `NewSemanticChunker` в `pkg/draftrag/semantic_chunker.go`
- Новая внутренняя реализация в `internal/infrastructure/chunker/semantic.go`
- Конфигурация: `SemanticChunkerOptions` с полями `Embedder`, `SimilarityThreshold`, `MinChunkSize`, `MaxChunkSize`
- Разбиение на предложения: rule-based (по `.` `!` `?` с защитой от сокращений)
- Вычисление эмбеддингов через интерфейс `domain.Embedder`
- YAML-конфигурация через `ChunkerConfig` (тип `"semantic"`)
- Включение в `NewPipelineFromConfig` и `PipelineOptions.Chunker`

## Контекст

- Существующий Chunker в `internal/domain/interfaces.go` не требует изменений — реализуется тот же интерфейс.
- SemanticChunker зависит от Embedder, в отличие от BasicChunker. Это новая связанность: чанкер требует вызова внешней нейросети.
- Sentence boundary detection не обязан быть идеальным для английского — достаточно rule-based; мультиязычность не требуется.
- Все публичные операции принимают `context.Context`.

## Зависимости

- `domain.Embedder` — интерфейс для вычисления эмбеддингов (уже существует)
- `domain.Chunker` — интерфейс, который реализуется (уже существует)
- Внешних библиотек для sentence splitting не требуется

## Требования

- RQ-001 Система ДОЛЖНА разбивать документ на предложения, используя rule-based sentence boundary detection (`.`, `!`, `?` со списком исключений: `Dr.`, `Mr.`, `Mrs.`, `Ms.`, `e.g.`, `i.e.`, `vs.`, `etc.` — после них точка не считается концом предложения).
- RQ-002 Система ДОЛЖНА вычислять эмбеддинг для каждого накопленного кандидата в чанк через переданный Embedder.
- RQ-003 Система ДОЛЖНА сравнивать косинусное сходство между эмбеддингом текущего кандидата и эмбеддингом предыдущего сформированного чанка.
- RQ-004 Система ДОЛЖНА начинать новый чанк, если сходство падает ниже `SimilarityThreshold` (0.0–1.0).
- RQ-005 Система ДОЛЖНА поддерживать `MinChunkSize` (в символах): не завершать чанк раньше достижения этого порога, даже при низком сходстве.
- RQ-006 Система ДОЛЖНА поддерживать `MaxChunkSize` (в символах): форсировать завершение чанка при превышении, на последней границе предложения.
- RQ-007 Система ДОЛЖНА уважать отмену `context.Context` и возвращать `ctx.Err()` при отмене.
- RQ-008 Система ДОЛЖНА возвращать ошибку `ErrInvalidChunkerConfig` при невалидных параметрах (порог вне [0,1], min < 0, max <= min).
- RQ-009 Система ДОЛЖНА поддерживать конфигурацию через `ChunkerConfig` с типом `"semantic"` в YAML-файле.

## Вне scope

- Замена или удаление BasicChunker — обе реализации сосуществуют
- LLM-based семантический чанкинг (использование LLM для определения границ)
- Мультиязычный sentence tokenizer с NLP-моделями
- Кэширование эмбеддингов предложений внутри чанкера (пользователь может обернуть Embedder в CachedEmbedder)
- Асинхронное/параллельное вычисление эмбеддингов для предложений одного документа
- Поддержка overlap (semantic chunker не имеет overlap — границы определяются семантически)

## Критерии приемки

### AC-001 Базовое семантическое разбиение на два тематических блока

- Почему это важно: основная ценность фичи — получение чанков, соответствующих разным темам.
- **Given** документ, содержащий два тематически различных абзаца (например, о погоде и о программировании) и SemanticChunker с `SimilarityThreshold=0.7`
- **When** вызывается `Chunk(ctx, doc)`
- **Then** возвращается не менее 2 чанков, и граница между ними совпадает с границей между абзацами
- Evidence: тест проверяет `len(chunks) >= 2` и `chunks[0].Content` не содержит предложений из второй темы

### AC-002 Влияние порога на количество чанков

- Почему это важно: порог — главный рычаг управления гранулярностью.
- **Given** тот же документ и два SemanticChunker с разными порогами (0.3 и 0.9)
- **When** вызывается `Chunk(ctx, doc)` для обоих
- **Then** чанкер с порогом 0.9 возвращает больше чанков (или столько же), чем чанкер с порогом 0.3
- Evidence: `len(chunksHigh) >= len(chunksLow)`

### AC-003 Уважение MinChunkSize

- Почему это важно: предотвращает слишком мелкие чанки.
- **Given** документ с тремя предложениями, где второе тематически отличается, и `MinChunkSize` больше длины одного предложения
- **When** вызывается `Chunk(ctx, doc)`
- **Then** второе предложение не образует отдельный чанк (присоединяется к первому или третьему)
- Evidence: `len(chunks) < 3`

### AC-004 Уважение MaxChunkSize

- Почему это важно: предотвращает слишком крупные чанки, превышающие лимиты модели.
- **Given** документ, где предложения семантически связны, но общая длина превышает `MaxChunkSize`
- **When** вызывается `Chunk(ctx, doc)`
- **Then** каждый чанк не превышает `MaxChunkSize` (по Content), и границы проходят по концу предложения
- Evidence: `all(len(c.Content) <= MaxChunkSize for c in chunks)` и каждый чанк заканчивается на `.`/`!`/`?`

### AC-005 Границы чанков не режут предложения

- Почему это важно: сохранение целостности предложений для качества эмбеддингов и ответов.
- **Given** произвольный документ
- **When** вызывается `Chunk(ctx, doc)`
- **Then** ни один чанк не начинается и не заканчивается в середине предложения
- Evidence: тест проверяет, что каждый `chunk.Content` заканчивается на `.`/`!`/`?` (за исключением последнего чанка, если документ не заканчивается знаком препинания)

### AC-006 Отмена контекста

- Почему это важно: согласованность с контрактом всех публичных операций.
- **Given** SemanticChunker и отменённый `ctx`
- **When** вызывается `Chunk(ctx, doc)`
- **Then** возвращается `ctx.Err()`
- Evidence: `errors.Is(err, context.Canceled)` или `context.DeadlineExceeded`

### AC-007 Валидация конфигурации

- Почему это важно: предотвращает неосмысленные конфигурации, приводящие к бесконечным циклам или паникам.
- **Given** невалидные параметры: `SimilarityThreshold` вне [0,1] или `MinChunkSize` > `MaxChunkSize`
- **When** вызывается `Chunk(ctx, doc)`
- **Then** возвращается `ErrInvalidChunkerConfig`
- Evidence: `errors.Is(err, ErrInvalidChunkerConfig)`

### AC-008 Пустой документ

- Почему это важно: граничный случай, не должен приводить к панике.
- **Given** документ с пустым `Content`
- **When** вызывается `Chunk(ctx, doc)`
- **Then** возвращается пустой слайс без ошибки
- Evidence: `len(chunks) == 0 && err == nil`

### AC-009 Конфигурация через YAML

- Почему это важно: единообразие с существующей системой конфигурации.
- **Given** YAML-файл с `chunker: { type: "semantic", semantic: { threshold: 0.75, min_chunk_size: 100, max_chunk_size: 2000 } }` и `NewPipelineFromConfig`
- **When** pipeline создаётся
- **Then** `pipeline.options.Chunker` является SemanticChunker с соответствующими параметрами
- Evidence: тест проверяет, что тип `opts.Chunker` соответствует `*semanticChunker` через type assertion, и `Chunk(ctx, doc)` возвращает `len(chunks) > 1` для документа с двумя тематическими блоками

## Допущения

- Sentence boundary detection по `.` `!` `?` достаточен для английских текстов; сложные случаи (мистер, Dr., e.g.) обрабатываются минимальным списком исключений.
- Embedder, переданный в SemanticChunker, способен осмысленно эмбеддить короткие фрагменты (1–3 предложения).
- Косинусное сходство является адекватной мерой семантической близости для данной задачи.
- SemanticChunker не используется для документов с единственным предложением — в этом случае возвращается один чанк.
- Пользователь осознаёт, что семантический чанкинг требует вызовов Embedder и может быть медленнее BasicChunker.

## Критерии успеха

- SC-001 Для документа из 3+ тематических блоков (по 3–5 предложений) semantic chunker с порогом 0.7 возвращает 3–5 чанков, в каждом из которых преобладает одна тема.

## Краевые случаи

- Документ из одного предложения → один чанк
- Документ, где все предложения семантически идентичны (дубли) → один чанк (если не превышен MaxChunkSize)
- `MinChunkSize = 0` → минимальный размер не применяется
- `MaxChunkSize = 0` → максимальный размер не применяется
- Embedder возвращает ошибку → ошибка пробрасывается вызывающему
- Документ содержит только пробельные символы → пустой слайс без ошибки

## Открытые вопросы

- Нужно ли добавить опцию `BufferSimilarity` (сглаживание скользящим окном) для устойчивости к выбросам? Отложено: в MVP используется сравнение с previous chunk embedding.
- Как быть с очень длинными предложениями (50+ слов)? Они будут включены целиком, даже если превышают MaxChunkSize — sentence boundary detection не режет предложения.
