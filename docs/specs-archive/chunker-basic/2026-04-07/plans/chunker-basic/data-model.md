# BasicChunker для draftRAG — Модель данных

## Scope

- Связанные `AC-*`: `AC-001`, `AC-005`, `AC-006`
- Связанные `DEC-*`: `DEC-002`, `DEC-003`, `DEC-004`
- Изменения persisted data model не требуются: чанкер статлесс, возвращает вычисляемые `Chunk` значения.

## Сущности

### DM-001 BasicChunkerOptions (options struct)

- Назначение: конфигурация базового алгоритма чанкинга (размер, overlap, лимит).
- Источник истины: публичный options struct в `pkg/draftrag`.
- Инварианты:
  - `ChunkSize > 0`
  - `Overlap >= 0` и `Overlap < ChunkSize`
  - `MaxChunks >= 0` (`0` означает “без лимита”)
- Связанные `AC-*`: `AC-001`, `AC-005`, `AC-006`
- Связанные `DEC-*`: `DEC-002`, `DEC-003`
- Поля:
  - `ChunkSize` — `int`, required, размер чанка в рунах.
  - `Overlap` — `int`, optional, default `0`, перекрытие между чанками в рунах.
  - `MaxChunks` — `int`, optional, default `0`, максимальное количество возвращаемых чанков.
- Жизненный цикл:
  - создаётся пользователем при создании chunker через фабрику.
  - не обновляется внутренне (immutable-by-convention).
- Замечания по консистентности:
  - недопустимы конфигурации, приводящие к неограниченному циклу (`Overlap >= ChunkSize`).

### DM-002 Chunk output contract (вычисляемые поля результата)

- Назначение: формат возвращаемых чанков для дальнейшего индексирования.
- Источник истины: доменная модель `domain.Chunk` + контракт в spec/plan.
- Инварианты:
  - `ParentID == Document.ID`
  - `Position` — 0..n-1 без “дыр”
  - `ID` детерминирован и зависит от `Document.ID` и `Position`
  - `Content` непустой (после `TrimSpace`)
- Связанные `AC-*`: `AC-002`, `AC-003`, `AC-006`
- Связанные `DEC-*`: `DEC-004`
- Поля (используемые в этой фиче):
  - `ID` — `string`, required, `fmt.Sprintf("%s:%d", doc.ID, position)`.
  - `ParentID` — `string`, required, равен `doc.ID`.
  - `Position` — `int`, required, индекс чанка в последовательности.
  - `Content` — `string`, required, фрагмент `Document.Content` по рунам.
  - `Embedding` — `[]float64`, not set (остаётся пустым; заполняется позже embedder’ом).
- Жизненный цикл:
  - создаётся при вызове `Chunk(ctx, doc)`.
  - не хранится (не persisted) и не обновляется chunker’ом.
- Замечания по консистентности:
  - недопустимо возвращать пустой `Content` или неконсистентные `Position`.

## Связи

- `DM-001 -> DM-002`: `BasicChunkerOptions` определяет форму `Chunk` результата (количество/границы/лимит).

## Производные правила

- Следующий `start` вычисляется как `end - Overlap` (или `end`, если overlap=0), при этом `start` должен монотонно расти.
- Если `MaxChunks > 0`, добавление чанков прекращается после достижения лимита (best-effort обрезка).

## Переходы состояний

- Отдельные переходы состояний не требуются: сущности статлесс/вычисляемые.

## Вне scope

- Persisted метаданные чанков, дедупликация, хеши контента.
- Токен-ориентированные ограничения (LLM tokens).
